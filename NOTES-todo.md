## T1 · 翻译按钮与相邻按钮视觉粘连,工具栏被撑宽

**状态**:✅ 已解决(2026-07-21)

**解决方式**:把工具栏从三组改为两组。

iOS 26 的 Liquid Glass 工具栏按 flexibleSpace 把按钮切成若干玻璃胶囊,
剩余宽度由弹性间隔平分。上游原本三组
`[已读 星标] | [下一篇未读] | [阅读视图 分享]`,我们加入第 6 个按钮后,
两个间隔各自只剩 40pt 出头,相邻胶囊边缘几乎相接 —— Liquid Glass 会把
靠近的玻璃融合,视觉上就是粘成水滴。

改为两组后剩余宽度全部给中间一个间隔,分隔清晰:
`[已读 星标 下一篇未读] ⟷ [阅读视图 分享 翻译]`
(左=这篇文章的状态与去向,右=拿这篇文章做点什么)

实现位于本 fork 的 `installTranslationButton()`,只在 iOS 26 分支重排,
旧系统分支保持上游的均匀分布布局。已截图确认两个胶囊分离。

## T4 · 离开文章再返回,译文丢失,需要重新翻译

**状态**:✅ 已解决(2026-07-19,本地缓存落地)
**发现时间**:2026-07-19,Phase 2 验收后由用户提出

**解决方式**:新增 `Shared/Translation/TranslationCache.swift`。
译文按「文章 ID + 模型 + 原文哈希」缓存在内存 + 磁盘(系统 Caches 目录,上限 50 篇,删最旧)。
再次打开已翻译过的文章并点翻译按钮 → 命中缓存,秒开、零请求。

设计要点:
- 只缓存**全部成功**的翻译;中途取消或有失败组的不缓存,避免固化残缺版本
- 换模型会重新翻译(键里含模型名,有意的 —— 便于对比模型效果)
- 文章内容更新后键会变,不会拿旧译文冒充新内容
- 未触碰 Modules/ArticlesDatabase(C 级禁区),缓存文件与上游数据库零交集

### 现象

翻译文章 A → 跳到上一篇/下一篇 → 再返回 A → A 显示原文,按钮为"未翻译"。

### 这不是 bug

译文存在于**网页的 DOM 里**(路线 B 的做法)。返回文章时 NetNewsWire 会重新渲染页面,
新页面是全新的 JS 环境,译文自然不存在。按钮显示"未翻译"与页面实际内容**一致**,
没有状态错乱。

但这也不是刻意的产品设计 —— 是路线 B 的固有后果,选型时未预先说明。

### 为什么现在不处理

用 mock 时重翻只要 1 秒,无感。接入真实后端后,每次返回 = 一次真实翻译请求,
才会变成实际问题(耗时、可能产生费用)。

**选哪个方案取决于后端的真实速度与成本,而这些数字目前不存在。**

### 三个候选方案

- **A(默认)**:什么都不做,每次重新请求。若后端自带缓存则第二次很快,
  完全符合 CLAUDE.md 第 5 节「缓存逻辑全部在后端」。**零代码改动。**
- **B**:Swift 侧内存缓存译文,返回时直接重新应用。体验最好,
  但在 Swift 侧存译文与第 5 节有擦边,需要用户确认。
- **C**:只记住"这篇翻译过"的标记,返回时自动重新请求一次。
  体验上像"记住了",但仍有网络开销。

**Phase 3 后端跑通、拿到真实耗时数据后再决定。**

## T5 · 个别组的翻译耗时呈数倍波动(尾部延迟)

**状态**:方案 1(provider throughput)+ 方案 2(对冲请求)均已上线,**观察效果中**(2026-07-22)
**严重程度**:体验问题 —— 一个慢组会卡住顺序阅读

### 现象(用户实测日志,同一篇文章、同一模型)

```
组4 第1次:完成,耗时 22.2s,原文 4326 字符
组5 第1次:完成,耗时 81.6s,原文 4374 字符   ← 尺寸相同,耗时 4 倍
```

### 数据能排除什么

- 不是切分问题:渐进分组已生效(组1=1243字符/4.1s 最先回,顺序基本正确)
- 不是模型本身性能:组4 和组5 尺寸几乎一样,同模型,耗时差 4 倍 → 是**服务端方差**
  (OpenRouter 会把请求路由到不同 provider,各家速度差异巨大,还有排队)

### 候选方案(按性价比排序)

1. ✅ **已实施(2026-07-19)**:请求体里加 provider 偏好 `"provider": {"sort": "throughput"}` ——
   让 OpenRouter 优先路由到吞吐量高的服务商(只对 openrouter 域名发送,其他服务商不认识此字段)。
   **观察中:用户日志里 80s 级异常值是否消失。**
2. ✅ **已实施(2026-07-22,用户仍报长文开头等 >30s 后上线)**:对冲请求(hedging)。
   先发一份,某请求超过「按大小估的健康耗时 × 2」还没成功,就并发补发一份,
   谁先成功用谁、另一份取消(`translate()` 用支持取消的 `URLSession.data(for:)`,
   输的那份能真掐掉,不白烧 token)。实现在 `TranslationController.hedgedTranslate`。
   **先导块用固定 4s 阈值(它在关键路径上,不回来后面都不开始);
   正文各组按大小估阈值(200 字符/秒 × 2 倍,下限 6s)。**
   只有真正偏慢的请求会被对冲,健康的不会平白多花钱。
   日志里 `对冲触发` 可看它多久触发一次(= 成本)。**观察中:开头是否稳定快进几秒内。**
3. 缩小 maxGroupChars(4000→3000):治标,收益有限,先不动。

### 相关缓解(2026-07-19 同轮实施)

断点续翻上线后,慢组卡住时用户翻页离开的代价大幅降低 ——
已翻好的组存成"未完成缓存",回来点一下接着翻,不重复花钱。

## T9 · 图标跟随深浅色切换 —— ✅ 没有问题,是我一开始漏看了一个开关

**状态**:✅ 已澄清(2026-07-21),**功能完全正常,无需任何改动**
**经过**:我先给出过一个**错误结论**,由用户指出后纠正

### 真相

主屏「自定」面板里,选中「深色」之后**下面还有一排子选项:「始终 / 自动」**。

- **始终** = 永远用深色图标,不管系统明暗
- **自动** = 跟随系统明暗切换 ← **这才是"图标会跟着主题实时变"的开关**

选「自动」后实测:系统切深色 → NetNewsWire 立刻变成黑底米色 R;
切回浅色 → 立刻变回浅色版。Apple 自己的「通讯录」「文件」同步切换。
**我们的三套素材工作完全正常。**

### 我一开始错在哪(值得记住)

我的错误结论是"模拟器不支持自动切换"。当时的推理是:
"Apple 自己的图标在深色模式下也没变 → 别人家也不跟随 → 不是我们的问题 → 是模拟器的问题"。

**前半段没错,后半段是错的。** 那条证据只能排除"我们写坏了",
**排除不了"有个全局开关没打开"** —— 因为那个开关(当时是「默认」)
本来就作用于**所有** app,包括 Apple 自己的。
两种假设对这条证据的预测**完全相同**,所以它根本没有区分能力。

方法上的教训已记入 L27。

### 顺带记下的正确操作

主屏长按空白处 → 左上角「编辑」→「自定」→ 选「深色」→ 再选下面那排的「自动」。
**只选「深色」不够,默认可能落在「始终」上。**

## T11 · 阅读视图:哪些源能抽出全文,哪些不能

**状态**:已实测一轮,列在下面;遇到新的失败源就往表里加
**记录时间**:2026-07-21

### 实测结果(2026-07-21,模拟器)

| 源 | 结果 | 备注 |
|---|---|---|
| 纽约时报中文网 | ✅ | 摘要 → 全文,约 3 秒。**最典型的受益场景** |
| BBC Chinese | ✅ | 成功,会带一点页面元信息(作者/发布时间/阅读时长) |
| 少数派 | ✅ | 成功,小标题层级都保留了 |
| Daring Fireball | ✅ | 链接博客,抽的是它指向的原文(Stratechery)—— 正是想要的行为 |
| **YouTube 类的源** | ❌ | **抽不出来,而且永远抽不出来** |

### YouTube 源为什么必然失败(不是 bug)

用户订阅里有好几个 YouTube 源(老孫聊遊戲、林亦LYi、石投记、Terry Chen 泰瑞…)。
查过数据库确认:这些文章的 `url` 是 `https://www.youtube.com/watch?v=...`,
`contentHTML` 长度为 **0**。

Readability 是从网页里找**文章正文**的。YouTube 播放页里没有文章,只有一个播放器 ——
**没有东西可抽**。点了会变成错误状态,这是诚实的结果,不是坏了。

原来那个 Feedbin/Mercury 服务同样抽不出来。

### 还没遇到但可能出现的失败模式

**前端渲染的站点**:如果页面在 `didFinish` 之后才用 JS 把正文渲染出来,
我们可能抓到一个空壳。对策是"抽到空就等 1~2 秒再试一次"。
**目前没有实测到这种情况,所以没有加这段重试逻辑**(不给没见过的问题写代码)。
用户如果报告某个源明明是文章却抽不出来,先怀疑这个,再加重试。

### 怎么排查

失败会记进日志(error 级别,会落盘):

```bash
xcrun simctl spawn booted log show --last 30m --predicate 'process == "NetNewsWire"' --style compact | grep 阅读视图
```

会打印失败原因和文章链接。

## T12 · Substack 的图片:多余按钮与被劫持的点击

**状态**:✅ 已解决(2026-07-21)
**用户报告**:"很多文章的图片下面都会出现这两个按钮,这是什么?"

### 那两个按钮是什么(查证过程值得复用)

**不是 iOS 的,也不是我们的代码** —— 是 **Substack 塞在 RSS 内容里的**。
直接把那篇文章的 `contentHTML` 从数据库拉出来就看到了:

```html
<div class="image-link-expand">
  <button class="… restack-image">   ← Substack 的「Restack(转发)」
  <button class="… view-image">      ← Substack 的「放大看图」
```

- 灰色空壳的那个是 Restack。它的图标用了 `stroke="var(--color-fg-primary)"` ——
  **这个 CSS 变量只存在于 Substack 网站**,在我们这儿没有,所以描边没颜色,渲染成空壳
- 蓝色箭头是放大按钮,用 `currentColor`,继承链接色所以显蓝
- **两个都是死的**:Substack 的 JS 不在我们这儿,点了没反应

### 顺带查出的更要紧的问题

Substack 把图片包在 `<a href="图片地址">` 里。
实测点图片 → **打开内置浏览器跳到 substackcdn.com**。

而 app 本来就有全屏图片查看器(`main_ios.js` 监听图片点击 → 原生查看器),
只是**被这个链接抢走了点击**,在 Substack 文章里永远触发不到。

### 解决方式(全在 `Shared/Appearance/nnw_appearance.js`,**上游零改动**)

1. CSS 一条 `.image-link-expand { display: none }` 藏掉两个按钮
2. 捕获阶段拦一道点击:点的是图片就 `preventDefault()`,掐掉跳转。
   **不改 DOM**,只取消默认行为 —— `main_ios.js` 照常收到事件,全屏查看器正常打开

### 长按:一度做了,又撤了

原计划自己做一个长按菜单(拷贝地址/浏览器打开/存相册),**代码写完并跑通了**。
但用户验收时发现 **WebKit 自带的长按菜单本来就有**
(打开链接 / 添加到阅读列表 / 拷贝链接 / 共享 / 保存到"照片" / 显示文本),
明确表示够用,于是**整段撤掉**。

⚠️ 注意其中的因果:那个系统菜单**正是因为图片被 `<a>` 包着才有的**。
我们只取消了跳转、**没有拆掉那个链接**,所以系统菜单照常工作。
以后如果有人想"顺手把这些多余的 `<a>` 拆了",会把这个菜单一起弄没 —— 别拆。

**最终的交互分工**:点一下 = 全屏看图;长按 = 系统菜单。

**本轮的处理方式**:大字号档位下**竖着堆**(favicon → 源名 → 标题 → 正文 → 时间),

`MainTimelineCellLayout.swift` 里有**两套**布局:

| 结构 | 何时生效 |
|---|---|
| `MainTimelineDefaultCellLayout` | 平时 |
| `MainTimelineAccessibilityCellLayout` | 系统字号调到「辅助功能」档位时 |

- 但如果将来要改的是**结构**(挪位置、增删元素),必须两套都改,并且实测:
  模拟器里 `设置 → 辅助功能 → 显示与文字大小 → 更大文字` 拉到最大,再看列表。

## T18 · 长期安装与分享给朋友:选项与一处我说错的地方

**记录时间**:2026-07-21

### ⚠️ 先纠正我说错的一条(重要,别再重复)

我曾告诉用户「这个 app 一次占 4 个 App ID,免费账号 7 天限 10 个,

那个「7 天 10 个」的额度管的是**新建** App ID。重装的是同样这 4 个,**不消耗额度**。

### 因此:删三个扩展 = 不划算,已决定不做

三个扩展是通过 app target 的「嵌入扩展」构建阶段挂进去的,

`NetNewsWire` 出现 143 处,扩展条目是结构性的,删了严重影响 merge。
收益只剩「编译快一点、包小一点」,不值得。

### 长期安装(自己用)

| 办法 | 说明 |
|---|---|
| 手动重装 | 每周连线 ⌘R,一分钟 |

## T19 · 同一篇文章「订阅源版」与「阅读模式版」的译文共用一个缓存键,会互相顶掉

**状态**:已知小限制,**不是 bug**,暂不处理(2026-07-22,item③ 落地时记录)

翻译缓存的键是「文章ID + 模型 + 提示词版本」,**不含「是不是阅读模式」**。
而阅读模式抽出来的正文和订阅源自带正文文字不同 → 存进条目里的 `bodyHash` 不同。
所以:先翻订阅源版、再翻阅读模式版,同一个键只留得下一份 ——
**两版会互相顶掉**,来回切时另一版要重翻一次(花一次钱)。

- **正确性没问题**:`bodyHash` 会把对不上的老译文挡掉,不会张冠李戴,只是白重翻一次。
- item③ 自动恢复不受影响:因为自动恢复只在「本地有匹配缓存」时才应用,
  对不上就停在原文等用户点(用户选的「不悄悄花钱」)。
- **要修的话**:缓存键里加一个「变体」维度(feed / reader),让两版能并存。
  改动都在我们自己的 `TranslationCache.swift` / `TranslationController.swift`,不碰禁区。
  但收益只是「来回切阅读模式时少重翻一次」,**除非用户实际觉得烦,否则不值得**。
