**Phase B 做了什么**(移动:多选「移动到…」为主,拖拽为辅)—— **待用户验收**:

| 能力 | 做法 |
|---|---|
| 编辑(多选)模式 | 右上角「编辑」进入,每个**源**前面出现勾选圈 |
| 移动到… | 底部工具栏,显示「移动 N 项到…」,弹出目标列表(各文件夹 + 「不放在文件夹里」) |
| 拖拽(辅助) | 拖源**放到文件夹行上** = 移进去;拖源**放到顶层的源上** = 拿出文件夹 |

**几个刻意的设计,别当成漏做**:
- **编辑模式下文件夹不可勾选**:Phase B 只移动源,而且文件夹那行还要留着点开/收起。
  (批量删文件夹是 Phase C 的事)
- **编辑模式下左上角「完成」会消失**:那个是"关页面",右上角的「完成」是"退出多选",
  两个都叫完成、意思还不同,并排必点错。
- **跨账户移动明确不做**:不同账户背后是不同同步服务,"移动"在那边没有对应语义,
  真要挪只能在目标账户重新订阅。选中跨账户的源会弹提示,不会静默失败。
- **拖拽只认两种落点**,拖到"文件夹里的源"上不接受 —— 那个落点很含糊
  (想进这个文件夹?还是排序?),与其猜错不如不接。也**没做拖到屏幕边缘自动滚动**:
  拖着一个源满屏找文件夹本来就不该是主路径,做多了反而鼓励用户走难走的路。
- **移动是串行的**:它们改的是同一批容器,并发搬容易互相踩;同步账户下串行还能
  避免把对方服务器打出限流(L33)。本地账户瞬时完成,串行不慢。
- **失败会列出具体是哪几个源**,不是笼统一句"移动失败"——否则用户不知道该重试哪个。

**离线验证(L63 的纪律:我点不了模拟器,就把逻辑抽出来跑)**:
`planMoves` 的决策表 8 个场景全过 —— 顶层→文件夹、文件夹→文件夹、文件夹→顶层、
两种原地移动被正确跳过、批量里混着原地的只搬该搬的、跨账户守卫拦得住也不误拦。

**待用户验收**:①右上角「编辑」→ 源前面出现勾选圈、文件夹没有;②勾几个 →
底部「移动 N 项到…」→ 选个文件夹 → 源确实进去了;③勾文件夹里的源 → 移动到
「不放在文件夹里」→ 源回到顶层;④长按拖一个源到文件夹行上;⑤退出编辑后底部工具栏消失。

**🔧 用户验收 Phase B 后报的问题,已修(2026-07-23)**:
「编辑模式下展开最后一个文件夹,里面的源和不在文件夹里的源缩进一模一样,分不清」。

**原因**:原来靠**缩进**区分这两种源,而一进编辑模式,每行前面要插一个勾选圈、
内容整体右移,**恰好把那点缩进差吃掉了**。

**修法:从结构上分开,不靠缩进。** 每个账户现在拆成两个分组 ——
上面「(账户名)」放文件夹,下面「不在文件夹里」放散源。不管在不在编辑模式都一目了然,
顺便给还没归类的源一个明确的名分(以后批量整理时一眼能看出哪些没归档)。

细节:**空分组不显示**(否则会出现只有标题、底下什么都没有的空壳);
一个文件夹都没有的账户,下半组的标题直接写账户名 —— 那时候写"不在文件夹里"是废话。

⚠️ **连带的一个已知限制**:拖拽"拿出文件夹"是靠**把源拖到某个散源上**实现的,

### 🚧 Phase C:批量删除 + 删文件夹释放源(2026-07-23,**待用户验收**)

**改动只有一个文件**:`FolderManagerViewController.swift`。**上游零改动**(删除复用上游现成的命令)。

| 能力 | 做法 |
|---|---|

**撤销是白拿的**:直接复用上游 `Shared/Commands/DeleteCommand.swift` ——
它本来就支持一次删多个 + 自己注册 UndoManager。我们只需要**搭一个它认得的节点父子关系**
(`makeNode(for:)`):父节点是文件夹 → 从那个文件夹里删;父节点是根 → 从账户顶层删。

**几个刻意的设计**:
- **编辑模式下文件夹的展开三角从 `.header` 换成 `.cell` 样式**:`.header` 是"点整行展开",
  而编辑模式下点整行的含义已变成"勾选",两者抢同一个点击必打架。
  换成 `.cell` 后:**点三角展开,点行选中**,各管各的。
- **勾了文件夹时「移动」按钮置灰**:文件夹没地方可搬(没有子文件夹),
  与其搬一半再报错,不如一开始就不让点。
- **空文件夹不问"里面的源怎么办"** —— 没有源要安置,问了是噪音。
- **释放时被单独勾中的源不搬**:用户既勾了文件夹(要释放里面的源)、又单独勾了其中某个源,
  那就是明确想删掉那一个 —— 不搬它,它随文件夹一起被删,正合其意。
- **搬运失败就停手不删**:半搬半删会留下一地无法解释的残局。会明确报告是哪几个源失败了,
  并说明"文件夹没有删除,免得连累里面的源"。

⚠️ **撤销的边界(重要,验收时别误会)**:撤销能盖住的只有「删除」这一步。
若刚才选的是"把源移到外面",**那个移动不在撤销范围内** ——
撤销后文件夹会回来,但里面是空的、源留在顶层。
(取舍:移动和删除是两个独立操作,硬凑成一个可撤销的整体不值当。)

**离线验证(L63 的纪律)**:7 个场景全过 ——
①三种删除各自会从哪个容器删(顶层源→账户 / 文件夹里的源→那个文件夹 / 文件夹→账户),
这条最要紧,搭错父子关系会**删错地方**;
②释放模式下该搬哪些源(只勾文件夹→全搬 / 连里面某个源也勾了→那个不搬 /
全勾→一个不搬 / 别的文件夹里的同名源不受影响)。

**待用户验收**:①勾几个源 → 红色「删除」→ 确认 → **摇一摇(模拟器 `⌃⌘Z`)看能否撤销**;

### 🆕 修:阅读模式下点翻译全篇先消失(2026-07-24,**待用户验收**)

**用户报**:Readability 拉过全文的状态下点翻译,整篇文章先全部消失,再流式进来。

**根因**:Readability 的输出外面包着**单子元素的壳**
(`<div id="readability-page-1"><div>正文…</div></div>`),而切分器的下钻条件写的是
「子元素 ≥ 2 个才下钻」—— 这层壳只有 1 个孩子,钻不进去 → **整篇文章成了一个组** →
流式开始要藏"第 0 组"原文,第 0 组就是全篇 → 全篇消失。
**副产品发现**:阅读模式的翻译从来没分过组(整篇一个请求,没有并行、没有渐进节奏)。

**修法三道**:
1. 下钻条件 `>= 2` → `>= 1`:递归会一层层剥壳,直到见到真正的段落。
   阅读模式从此和普通模式**行为一致**(先导块 750 + 渐大分组 + 并行),顺带变快
2. `streamLeadBegin` 兜底:第 0 组 > 5000 字符(正常 ~750)就拒绝流式,
   走非流式赛跑(原文一直显示、整块替换)—— "全篇被当成一组"的伤害太大,值得留一道闸
3. `promptGeneration` 5→6:阅读模式文章的组边界全变,旧未完成缓存必须作废

**待用户验收**:①阅读模式下点翻译 → 原文**不再消失**,只有开头一小段变成流式译文,
其余部分保持原文、随后逐块被替换;②阅读模式的翻译总耗时应明显缩短(有并行了);
③普通模式(非阅读视图)的翻译行为没被弄坏。

2. **导航栏是透明的,停靠时正文直接从标题背后穿过去(字叠字)** ——
   → 浮层自带一条纸色底,后半程淡入。
   ⚠️ **第一版在这里犯了错(已修,见 L56)**:我同时给 `standardAppearance` 也铺了纸色底,
   而**导航栏画在控制器 view 之上** —— 结果标题飞上去钻到它背后消失了,
   那条带子还把头图和正文一起挡住。现在导航栏**自始至终透明**,底只由浮层提供,
   层级是「正文 → scrim → 标题 → 导航栏按钮」。
   (仍要显式设透明:iOS 默认的 standardAppearance 滚动后会自己画一层毛玻璃。)
   ⚠️ 是 **navigationItem 上的** appearance,不是全局 appearance 代理 ——
   后者会把大标题和副标题一起冲掉(L45),上游自己在文章页也是这么做的。

**另一个坑**:停靠时显示的那个标题原本是**上游的 `navigationItem.titleView` 标签**。

### ✅ 修「翻译 API Key 一直显示已设置」+ 两个连带问题(2026-07-22,已验收)

用户报告:把已填的 API Key 删掉后,设置页那行**不管点勾还是点叉都还写着「已设置」**。
用户猜是「key 和服务地址填了一个就算已设置」——**不是**,判断只看 key。真原因是三条:

1. **主因:那行没被重算**(教训 L47)。「已设置 / 未设置」只在 `cellForRowAt` 里算,
   从子页 pop 回来时 cell 还在屏幕上、UIKit 不会重新问一遍,显示的是进去之前的旧文字。
   `SettingsViewController.viewWillAppear` 里更新了主题 / 配色等一堆 label,**唯独没有 `reloadData()`**。
   「翻译模型」「界面语言」两行有同样的毛病(值总是非空,所以不容易察觉)。
   → **修法:`viewWillAppear` 末尾加一行 `tableView.reloadData()`**(上游文件,方法末尾追加,带 `[翻译]` 标记)。
2. **「勾保存」改造的漏网之鱼**(同 L47)。`TranslationAPIKeyViewController.textFieldShouldReturn`
   里还留着 `save()` —— 按键盘的「完成」键就落库了,之后点「取消」根本取消不掉,
   **「点叉不保存」这个承诺一直是假的**。→ 删掉那句,落库只由右上角的勾负责。
   已 grep 确认其余设置子页没有同类残留(命中的 4 个都是上游账户页,A 级禁区,不碰)。
3. **顺带查出:服务地址的合法性判断形同虚设**(教训 L48)。原来用
   `chatCompletionsURL != nil`,而实测 `URL(string:)` 对 `abc`、`abc def`、
   `openrouter.ai/api/v1`(少了 https://)、空串**统统返回非 nil**(当成相对地址)。
   影响不只是显示 —— 服务地址写错时不报「地址不合法」,而是拿相对地址去发请求、
   报一个看不懂的网络错误。→ `chatCompletionsURL` 加上「scheme 必须是 http/https 且 host 非空」。
   **9 个用例回归**:默认 OpenRouter / 带尾斜杠 / `http://localhost:1234/v1` 自建 /
   别家兼容服务全部通过,5 种写错的全部拒绝(不误杀和拦得住一起量)。

**关于用户提的「必须全部填写完成才算已设置」**:没有原样照做,**照做会改出新 bug** ——
服务地址**留空是合法且推荐的**(留空 = 用默认 OpenRouter,页面底部说明里就写着),
要求两个框都非空会把「只填 key」这个最常见的正确配置误报成「未设置」。
改为接住用户的意图:新增 `TranslationConfigStore.isFullyConfigured`
(= `configurationProblem == nil`,即 **key 非空 且 服务地址能拼出合法请求地址**),
设置页那行改用它。

**改动文件**:`TranslationConfig.swift`(+`isFullyConfigured`、`chatCompletionsURL` 加校验)、
`TranslationAPIKeyViewController.swift`(去掉 return 键的 save)、
`SettingsViewController.swift`(上游,两处:`viewWillAppear` 末尾一行 + 判断换成 `isFullyConfigured`,净 +12/-1)。
双平台编译通过,已装模拟器。**用户三步验收全部通过(2026-07-22)**:
①删 key 点勾→当场变「未设置」 ②删 key 按键盘完成键再点叉→仍是「已设置」(即取消真的取消了)

| 以前的困局 | 真正的病根 | 修法 |
|---|---|---|

1. **纸色底从网页收归 UIKit**(`WebViewController.nnwUseUIKitPaperBackground`):

2. **scrollEdge 显式透明**(`ArticleViewController.nnwRefreshNavigationBarAppearance`):
   scrollEdge = 透明(顶部通透)、standard = nil(滚起来是系统毛玻璃)。
   **别再图省事全设 nil** —— 那样两态用的是同一套毛玻璃(L61)。
3. **把滚动视图指给导航栏**(`nnwTrackCurrentArticleScrolling` + `setContentScrollView`):
   系统靠它才知道"内容滚到哪了"。WKWebView 藏在翻页容器的子页面里,系统自己找不到。
   调用两处:网页加载完 + 翻页动画结束(翻到的页可能早加载完了,不会再触发前者)。

`headerDockedScrimFeather`(24,羽化高度)、`headerDockedScrimMaterial`(材质)。

### 🔥 修「文章内容页往下滑几秒后闪退」(2026-07-23,**我上一轮改动引入的,已修待验收**)

**现象**:用户在文章内容页往下滑,卡死几秒后闪退。崩溃报告里 **28000 层递归**。

**根因(从崩溃栈读出来的,不是猜的)—— 两个各自正确的功能咬合成了死循环**:
```
滚动 → 藏工具栏 → 导航栏收起 → 安全区变化 → 系统调整滚动位置
     → 又触发滚动回调 → 判断方向 → 显示工具栏 → 安全区又变 → …… 来回拉锯没有尽头
```
「滚动藏栏」上线时这个回路就已存在,只是闭合得松;**我这轮加的
`setContentScrollView`(让顶栏能做两态)使导航栏开始主动更新那个滚动视图的安全区**,
回路一下闭合得又快又紧。崩溃栈里
`_updateViewSafeAreaInsetsAndEagerlyUpdateContentScrollView` 这一帧就是铁证。

**修法**:给藏/现栏加**防重入闸门**(`isTogglingBars`),切换期间滚动回调一律原地返回;
闸门到下一轮 runloop 才抬起(切换引发的布局是本轮同步跑完的,要整段罩住);
抬闸时 `primed = false` 重新取基准。`hideBars`/`showBars` 收进唯一入口
`nnwToggleBars(hide:)` —— ⚠️ **以后别绕过它直接调那两个方法**。

**离线复现 + 验证(这次的关键补救,见 L63)**:把判断逻辑连同这条回路写成几十行模拟,

即导航栏+状态栏真实高度 + 快速滑动),**加闸门后 0 种、最深 2 层**。

### 🔁 拖拽整段推倒重写:从「压着哪一行」改为「落在哪个区域」(2026-07-23)

**用户一句话点破了根本分歧**:
>「不能只判定拖到了文件夹上,或者『不在文件夹里』这几个字上放开才算拖动放入,
> **拖动后在整个对应的区域放手,都应该算拖动放入**」

前两轮我一直在补"压着哪一行"这个模型的漏(先按行、再按组),补不完是因为**模型本身就错了**。
用户的心智模型是**区域**。重写后的三条规矩:

1. **落点先翻译成"要进哪个容器"**,由 `dropDestination(at:)` 一个函数说了算,
   **悬停判定和放手执行共用它** —— 从结构上杜绝"看着能放、松手没反应"
   (前两轮正是两处判断各写一遍、口径对不齐)。
2. **落在空白处不算失败**:取该位置**上方最近**的那一行,按它所属区域算。
   于是行与行之间、一组的末尾,全都有明确归属。
3. **区域归属看那一行是谁**:文件夹行 / **文件夹里的源** → 都算"那个文件夹"
   (所以展开的文件夹整片都是有效落点,这是用户明确要求的);顶层的源 → 算"账户顶层"。

**同轮修的另外两件**:
- **拖动时屏幕乱弹**(用户:「滚动后手一停,屏幕唰地弹回去,几乎无法准确操作」):
  真因是**拖动期间列表被重画了** —— 后台抓取来一条通知就整棵重建,重建会重置滚动位置。
  → 新增 `isDragInProgress`,拖动期间冻结刷新,拖完再补画一次。
- **拖向文件夹没有动画反馈**:原来两种落点用了不同的 intent ——
  拖到文件夹用 `.insertIntoDestinationIndexPath`(只把目标行轻微高亮,
  而我们给每行铺了自定义暖纸底色,那层高亮几乎看不出来)。
  → **统一改用 `.insertAtDestinationIndexPath`**,周围的行会让开一条缝,
  和用户称赞的"拖出文件夹时的互动效果"完全一致。

**离线验证**:8 个落点场景全过 —— 压在文件夹行上 / 压在展开后的子行上 / 压在最后一个子行上 /
压在收起的文件夹上 / 压在顶层源上 / 压在两个源之间 / **落在最后一个源下方的空白**
(前两轮就是死在这里)/ 落在所有内容上方(不猜,判无效)。

**是模拟自己报错才发现的** —— 顺带说明这类几何逻辑光靠脑补确实不可靠。

**② 松手动画"往右错位、卡一秒"。**
原因:预览落点用的是**手指松开的位置**,而手指通常不在行的水平正中 → 预览歪在右边停一下。
→ 改成落到**它现在真正待的那一行的正中**:此时数据已经重排完
(那段在 `defer` 里、排序之后才跑),直接问"这一项现在在第几行"就能拿到最终位置,预览一步到位。
跨容器搬家时查不到(源换了文件夹、身份变了),退回手指位置。

**🔧 修「按编辑后勾选圈有的出现、有的不出现」(2026-07-23,待验收)**:

**原因**:每行的外观只在"创建/重新配置"时才决定,而**进出编辑模式时数据本身没变** ——
原来这里调 `reloadFromAccounts`,它算出的快照和当前一模一样,
diffable 判定"无事发生",**屏幕上现有的行保持原样、不重新配置**(所以没有圈);
只有滚动之后新生成的行才带圈 → 于是就成了"有的有、有的没有"。

**两处修法**:
1. **勾选圈不再自己判断 `isEditing`**,固定写 `.multiselect(displayed: .whenEditing)` ——
   露不露面交给系统判断,比自己在每次进出编辑模式时重新配置可靠。
2. 但**文件夹的展开三角必须跟着编辑模式换样式**(`.header` 点整行展开 ↔ `.cell` 点三角展开,
   否则"点整行"会同时意味着展开和勾选),这一处躲不掉 →
   `setEditing` 里改用 **`collectionView.reconfigureItems(at:)`** 把可见行重跑一遍外观。
   它只重配外观、不动数据和层级,也不会像整棵重建那样闪一下。

**🔥 上面这个 reconfigure 又崩了一次,已修(2026-07-23,并入 L66)**:
用户报「把源从文件夹里拖出来时闪退」。栈直指
`performMove` 的收尾 → `setEditing(false)` → `reconfigureItems`。
原因:**退出编辑模式有时是"刚搬完源之后自动触发"的**,那一刻列表和数据源还没对齐,
`indexPathsForVisibleItems` 里混着已经不存在的行 → 撞 UIKit 断言。
→ 两道保险:**延到下一轮 runloop**(等数据稳定)+ **只挑数据源确实认得的行**。
⚠️ 这已是当天第二次「在数据刚变完的时刻调 UIKit 批量操作」而崩,
改这块时**默认就该问一句:我调用它的这一刻,列表和数据源一致吗?**

**🧹 最终结论:那次刷新已被整个拿掉(2026-07-23,待验收)**

崩到第二次之后没有继续加保险,而是回头问"它为什么存在" ——
答案只有一个:让文件夹的展开三角跟着编辑模式换样式(`.header` ↔ `.cell`)。

于是每行的外观**完全不依赖编辑状态**,进出编辑模式什么都不用刷 →
`reconfigureItems` 从代码里彻底消失,那条崩溃路径不复存在。
⚠️ **别再往 cell 的 accessories 里加 `isEditing` 条件** —— 一加就又需要手动刷新了。

### 🚧 拖到「两个文件夹之间的顶层」+ 排序逻辑合并(2026-07-23,待验收)

**用户场景**:「A 展开、B 收起,想把 A 里的一个源拖到 A 和 B 中间(不属于任何文件夹)」——
原来做不到,会被判成"放进 A"。

**这是一条本质有歧义的缝**:A 的最后一个子行和 B 之间,既可能是"放进 A 的末尾",
也可能是"放到 A 和 B 之间的顶层"。上下位置分不出来 → **改按水平位置分**:
文件夹里的源本来就比外面的源多缩进一段(`nestedFeedExtraIndent`),
**手指退到那条缩进线左边(x < `outdentThreshold` = 56)就算"拿出来"**。
Files / Finder 里挪动嵌套项目也是这个手感。

**顺带合并了两套几乎一样的排序代码**(`reorderTopLevel` / `reorder` → `reorderWithinLayer`)。
新的抽象是「层」:**顶层 = 文件夹和散源混排的一串;文件夹内 = 它自己那些源**。
两层共用一套算法,只是"这一层有哪些项""项的键怎么取"不同。

**合并时揪出一个原本就有的隐患**:搬进顶层时,插入位置**只按散源算**,
可是顶层的真实次序是"文件夹和散源混排"的 —— 所以从文件夹里拖到两个文件夹之间,
即使方向对了也会排到所有散源的最后。现在统一按整层次序算。

**离线验证 9 个场景全过**,头两条就是用户那个场景
(落点是 A 的最后一个子行 → 插到 A 之后 → a2 正好落在 A 和 B 中间);
另外覆盖:同一落点但目标是留在 A 里、落点是文件夹/散源、末尾空白、拖文件夹、层的隔离。

---

**(以下是先前那轮的结论,前提仍然成立,只是不再是终点)**
**上游模型确实不存顺序(详见 L65)**:
`Folder.topLevelFeeds` / `Account.topLevelFeeds` 都是 **`Set<Feed>`,本身无序**;
上游列表的顺序是显示时**现算**的(`sortedAlphabeticallyWithFoldersAtEnd()`,
强制按字母排、文件夹在后)。所以就算管理页自己存一份顺序,
**主列表照样按字母排**,两处对不上;OPML 导出导入、同步账户也都不带这个顺序。
**上游的设计意图是"用名字控制顺序"** —— 用户自己的文件夹就是这么起名的
(`01 宏观`、`02 社会`…),源要排序照做即可。

### 🎯 拖放落点引入「上下边缘带」(2026-07-23 晚,**待用户验收**)

**用户一句话否掉了上一版方案**:
>「A 展开、B 收起,把 A 的源拖到 A B 中间这个场景**几乎很难触发** ——
> 因为我想拖到中间,就**必然会先触发 B 展开**,然后就没办法拖到中间了。
> **这是一个写在了代码里,但几乎没机会触发的场景。**」

**核查后属实。** 上一版能落到「A 和 B 之间的顶层」的,只有一块区域:
A 展开后的**最后一个子行** + 手指**靠左**(x < 56)。而 B 那一行**不管手指在左在右
一律判成"放进 B"**,并且停 0.6 秒就弹簧加载展开。所以用户瞄那条缝时,
手指一进 B 的行就变成"放进 B" + 自动展开 —— 可用的目标区只剩分界线上方那半行,
既看不见也没提示。

**根本原因:整套判定只有一个维度** —— 手指压着哪一行。行与行之间那条缝算什么,
全靠"往上取最近的一行"猜,而猜出来的结果**表达不了"同层插队"这件事**。

**改法:加第二个维度 —— 手指落在那一行的哪一段(纵向分带)。**
这是 Finder / Files / 各种大纲编辑器的通行手感:

| 手指在文件夹行的哪一段 | 意思 |
|---|---|
| **上边缘带** | 排在这个文件夹**前面**(顶层) |
| **中间带** | **放进**这个文件夹(← 只有这一段会触发悬停自动展开) |
| **下边缘带** | 排在这个文件夹**后面**(顶层) |

于是「拖到 A 和 B 中间」= 瞄 **B 的上边缘**,那是一条肉眼看得见的分界线,
而且**停在那儿不会让 B 弹开**(边缘带不计时)。用户原来的痛点直接消失。

**顺带白拿一个原来做不到的能力**:把源/文件夹**排到某个文件夹的前面**。
以前顶层混排里只能压在别的散源上凑,现在文件夹本身也能当插入锚点。

**两个刻意的设计**:
- **展开着的文件夹,下边缘带并入"放进去"**:那一带紧挨着它的第一个子行,
  屏幕上看就是"文件夹里面";顶层意义上的"它后面"在整片子行的**下方**,不在这儿。
- **旧的水平规则(靠左 = 拿出文件夹)保留**:它解决的是另一个歧义(哪一层),
  和纵向分带各管各的。于是"拿出文件夹"现在有两条路,都通。

**新增两个文件,上游零改动**:

| 文件 | 作用 |
|---|---|
| `iOS/FolderManager/DropZoneResolver.swift` | 落点判定的**纯规则**(不碰 UIKit) |
| `tools/sim-dropzone.swift` | 离线跑决策表 |

**为什么要把规则单独拆出来**:拖放是我(AI)点不了的路径,而它又是本项目踩坑最多的地方。
拆出来之后离线脚本能把**真代码原样编译进去**跑,而不是抄一份副本对着跑(副本迟早长歪)。

```bash
swiftc -o /tmp/sim-dropzone \
  "iOS/FolderManager/DropZoneResolver.swift" "tools/sim-dropzone.swift" && /tmp/sim-dropzone
```

**离线验证 28 项全过**,其中最要紧的一条是**穷举 168 种组合验防崩不变量**:
「会触发文件夹展开」必须永远蕴含「落点不插占位」—— 这正是 L65 那次闪退的根因,
现在 `springLoadable` 是从 `isInsertInto` **推**出来的,不是另判一次,结构上不可能对不齐。

⚠️ **仍然验不了的**:`coordinator.drop` 调用顺序、批量更新时数据源一致性这类
UIKit 内部状态机问题,离线复现不出来(L66),只能靠实测。

**待用户验收**(iOS 编译通过、装机已核对二进制指纹一致):
1. **主场景**:A 展开、B 收起 → 拖 A 里的一个源 → 停在 **B 那一行的最上沿** →
   应看到 A 和 B 之间**让开一条缝**、且 **B 不会自己展开** → 松手,源落在 A、B 之间的顶层
2. **没弄坏原来的**:拖到 B 那一行**中间** → B 高亮 → 停 0.6 秒 B 照常自动展开 → 松手进 B
3. **新能力**:拖一个顶层的源 → 停在某个文件夹的**上沿** → 松手,它排到那个文件夹前面
4. **旧路径还在**:拖 A 里的源 → 停在 A 的子行上、手指**往左带一截** → 松手,源被拿到顶层
5. 拖文件夹调顺序、拖进/拖出文件夹,和之前一样正常

### 🎯 落点提示:「松手会进这个文件夹」看得见了(2026-07-23,**待用户验收**)

**用户报**:「拖到文件夹行中间松手会进那个文件夹 —— 但**某些时候我不知道我松手,
它是会落在两个文件夹中间,还是某个文件夹里面**。能否在放进去之前给个提示?」

**病根不是没有提示,是提示看不见。** 系统对两种落点本来就有区分:
「插到这个位置」= 周围的行让开一条缝;「放进这一项」= **目标行高亮**。
但那层系统高亮**很淡**,而我们给每一行都铺了自定义暖纸底色,
压上去几乎看不出来。(这个现象其实早在拖拽重写那轮就记过一笔,当时的处理是
"两种落点统一改用让位动画" —— 治标,反而把"进去 vs 插旁边"这个区别抹平了。)

**修法:自己画一层,给两条独立线索**:
- **底色**换成强调色(陶土红)的淡色版
- **图标**从空心 `folder` 换成实心 `folder.fill`

一色一形,哪种屏幕条件下都至少认得出一个。

**浓度是算出来的,不是拍脑袋**(我点不了模拟器,但色差可以直接算):
把强调色按不同浓度叠在暖纸底上,和普通行比每通道差值 ——
0.18 → 浅色下 (9,26,32);0.22 → (11,32,39);**0.28 → (14,40,50)**。
前两档在浅色下偏弱,而这是个**转瞬即逝**的提示,宁可过一点也别看不见,所以取 0.28。

**为什么这个能做,而"悬停自动展开/合上"做不了**(同一天里的对照):
落点提示**只改已经存在的那两行 cell 的颜色和图标**,不增删行、不动数据源 ——
所以完全不受"拖动途中不能改数据源"那条约束的限制。
**"改外观"和"改结构"在拖动途中是两种完全不同的风险。**

**高亮的条件和落点意图出自同一个 `decision`**,所以"看到的"和"放手的结果"必然一致 ——
这一页从落点重写起就守着这条规矩(见 `dropDecision` 的注释)。

**待用户验收**:①拖一个源到文件夹行**中间** → 那一行**整行变色 + 图标变实心** →
松手确实进去了;②手指移到**上/下边缘** → 高亮**消失**、改为出现一条缝 →
松手落在文件夹外面;③手指拖出列表范围 / 取消拖动 → 高亮**不会留在屏幕上**;
④深色模式下同样看得清。

> 如果觉得还不够醒目,下一步可以加个淡入或轻微脉动 —— 但**拖动时的提示我倾向于瞬时切换**,
> 手指在动的时候,渐变反而让人分不清"现在到底选中的是哪一行"。

### 🗑️ 「悬停自动展开文件夹」整个拿掉了(2026-07-23,**最终结论,别再加回来**)

**用户第二次说明后才听懂**:问题不在"跨多次拖动累积",而在**同一次拖动之内** ——
>「悬停在一个文件夹上,它会展开,但如果我没有放开、而是把拖动的东西移开,**文件夹不会合上**。
> 这就导致列表变得很长,如果我想找靠下的一个位置,**就会需要悬停滑动很长一段距离才能放下**。」

**而"手指一离开就合上"是做不到的**(硬约束,不是偷懒):合上要删行,
而手指刚离开文件夹之后的落点**几乎都是 `.insertAt`(带占位空隙)**,
那一刻改数据源就撞 UIKit 批量更新校验、直接崩(L65 已崩过一次)。
唯一安全的时机是"正悬停在另一个文件夹中间"和"拖动全部结束",**恰恰都不是用户要的时机**。

**于是给了用户三个选项,用户选了「拿掉」**:

| 选项 | 用户的选择 |
|---|---|
| ① **拿掉悬停自动展开** | ✅ **选它** |
| ② 拖动一开始先全部收起、拖完恢复 | ❌ |
| ③ 硬做"一离开就合上"(两步撤缝法,有崩溃风险) | ❌ |

**这个机制存在了大约一天,期间给用户添了两次麻烦、让 app 崩了一次**:
挡住「A 和 B 之间」那条缝(用户:「几乎没机会触发」)、把列表越撑越长、L65 那次闪退。
按 L66 的原则:**一个需要不断打补丁才能不崩、不别扭的机制,通常说明它本来就不该存在。**

**拿掉之后白拿的三件事**:
- **拖动全程不再修改数据源** → **L65 那条崩溃路径从代码里消失了**
- 拖动期间**列表长度恒定** → 落点不会在手指底下移动
- 代码净减少(删掉约 110 行:计时器、自动展开集合、展开/合上的快照操作、拖动结束的收尾)

**代价(明知的取舍)**:想把源放进文件夹**里面的指定位置**,要**先点开那个文件夹再拖**。
只是"丢进某个文件夹"不受影响 —— 拖到文件夹行上松手即可,和以前一样。

**离线验证同步更新**:原来那条防崩不变量(「会触发展开 ⟹ 落点不插占位」)已随机制一起删除,
换成「『放进这一项』的意图只出现在目标确实是那个文件夹时」。
⚠️ **若有人把弹簧加载加回来,必须把那条防崩不变量一并加回来**(sim 脚本里写了这句提醒)。
27 项检查 + 168 种组合穷举全过。

**待用户验收**:①拖一个源在收起的文件夹上悬停几秒 → **它不再自己展开**,列表长度不变;
②拖到文件夹行**中间**松手 → 源照样进到那个文件夹里;
③先点开文件夹 → 再拖一个源进去 → 能放到文件夹**里面的指定位置**;
④整段拖动过程中,列表不会在手指底下变长变短。

### 🌀 弹簧加载展开的文件夹,拖完自动合上(2026-07-23,**已作废:整个机制已拿掉**)

**用户提的**:
>「拖动一个源在文件夹上悬停、文件夹打开了,能否让我从文件夹上悬停后拖走的时候,
> **文件夹自动再合上**?只开不合的话,整理订阅源时**整个列表越变越长**,
> 想找到一个精确的目标位置就变得路程很远。」

**做了两条**:
1. **同一时刻最多只有一个「自动展开」的文件夹** —— 弹簧加载展开新的时,把上一个合回去
2. **拖动结束时,自动展开过、又没被用上的文件夹全部合回去**

**⚠️ 做不到的那半条,以及为什么(重要,别当成漏做)**:
**「手指一离开就立刻合上」不能做。** 合上会删掉若干行,
而拖动途中列表里常常存在一个 UIKit 插的**占位空隙**(就是那条"让开的缝",
落点意图为 `.insertAt` 时才有)。占位不在数据源快照里,此时改数据源
→ UIKit 校验行数对不上 → **直接崩**(L65 为此崩过一次)。
而"手指刚离开 B"之后的落点**绝大多数正是** `.insertAt`。
所以合上的时机只能挑两个安全点:**展开另一个文件夹的那一刻**(那时用的是不插占位的意图)、
**拖动全部结束之后**。净效果是「拖动过程中最多多出一个文件夹的长度,一松手就复原」。

**两个刻意的设计**:
- **只合我们自己开的**:`autoExpandedFolders` 只记录本次拖动中**由弹簧加载**展开的文件夹,
  用户自己点开的绝不会进这个集合。已核对:往这个集合里加东西的地方**全代码只有一处**
  (弹簧加载那一处),所以这条保证是结构上成立的,不靠"小心别写错"。
- **源真的落进去的那个文件夹留着展开**:放手时就把它从集合里摘掉了。
  刚搬完就把结果藏起来,人会以为东西搬丢了。
- **合上的动作不在拖放回调里直接做**:只改自己的记录,让紧接着那次
  `scheduleReload` 顺手渲染成合上的样子 —— 拖放刚结束那一刻列表和数据源不一定已对齐,
  在那里直接 apply 属于"数据刚变完就调 UIKit 批量操作",当天为此崩过两次(L66)。

**待用户验收**:①拖一个源到收起的文件夹 B 上悬停 → B 展开 → **拖走、放到别处** →
松手后 **B 自己合上**;②拖着依次悬停 B、C、D → 同时最多只有一个是开着的;
③**把源真的放进 B** → B **保持展开**(能看到刚搬进去的那个);
④**你自己点开的文件夹**,拖动一圈之后**仍然是开着的**(不许被合上);
⑤反复拖十几次,列表长度不会一路变长。

```bash
UDID=A7B1AE1F-0391-4C3A-B979-FA35653256FF   # xcrun simctl list devices booted 可查
```

## 八、每次改完代码,装模拟器请用这个脚本

```bash
./tools/install-to-simulator.sh
```

**不要**直接 `xcrun simctl install`。原因(L41,2026-07-21 真实踩过):
`simctl install` 会把新版装进一个**新容器**,而系统注册的仍是旧容器 ——
命令返回成功、app 能启动,**但跑的是旧代码**。
本轮因此让用户测了 40 分钟前的二进制,还据此排查了一整轮无效功。

脚本装完必 `cmp` 比对,不一致就 rsync 覆盖到系统正在用的那个容器
(非破坏性,不动数据容器,订阅源和 Keychain 都不会丢)。
**实测这个卡死是持续存在的**,脚本每次都会报警并自动兜住。

排查「改了没生效」的第一步永远是:**先证明跑的确实是新代码**,再去查逻辑。
```bash
A=$(xcrun simctl get_app_container booted com.ranchero.NetNewsWire.iOS-DEBUG app)
```
