# NOTES-lessons.md — 经验教训

> 本项目踩过的坑和吃过的亏,按时间顺序编号。
> **接手的 AI 请先通读一遍** —— 这里每一条都是真实付出过代价的,
> 不读的话大概率会再付一次。
> 格式:现象 → 教训 → 怎么应用。

最后更新:2026-07-19

---

## L1 · 全新 clone 后第一次编译必失败

**现象**:`cannot find 'SecretKey' in scope`,两处报错都在 Feedly 相关文件。
**原因**:`SecretKey.swift` 被 .gitignore 排除,由编译期脚本生成,但生成时机晚于 Secrets 模块编译。先有鸡还是先有蛋,只在第一次出现。
**应用**:再编译一次就好。**绝对不要去"修"那两个报错的文件** —— 那是 A 级禁区,而且根本没坏。

## L2 · 仓库根目录的新文件夹不参与编译

**现象**:原计划把新代码放根目录 `Translation/`,但那里的代码 Xcode 根本看不见。
**原因**:本工程用 Xcode 16 的文件系统同步组机制,只有 `Mac/ iOS/ Shared/ Modules/ Widget/ Tests/ Technotes/ xcconfig/` 这 8 个文件夹里的新文件会自动进编译,根目录不在其中。
**应用**:所有新代码放 `Shared/Translation/`。**新增文件前先确认目标文件夹在同步组清单里**(grep pbxproj 搜 `PBXFileSystemSynchronizedRootGroup`)。

## L3 · 用探针文件实测,不要只靠推理

**现象**:「`Shared/Translation/` 下的 .js/.json 会不会被打进 app 包?」—— 光看工程文件推不出来。
**应用**:放一个一行的探针文件 → 编译 → 直接 `ls` 构建产物的 app 包看它在不在 → 删探针。**一次真实构建胜过十次推理**,本项目用这招验证过 3 次关键假设。

## L4 · NSLocalizedString 会污染上游的共享文件

**现象**:代码里用了 `NSLocalizedString`,编译后发现 `Shared/Localizable.xcstrings`(上游共用大文件)被 Xcode 自动塞进了我们的中文文案。
**教训**:这种"编译器自动改文件"的副作用,肉眼看代码根本发现不了,是 merge 冲突的隐形炸弹。
**应用**:本 fork **禁用 NSLocalizedString**,直接写中文字符串(单用户 app 不需要多语言)。每次编译后跑 `git status` 检查有没有意料之外的文件被改。

## L5 · 上游文件的 private 成员够不着 → 用"纯末尾追加"的扩展

**现象**:`webView`、`currentWebViewController`、`toolbarItems` 都是 private/内部的,`Shared/Translation/` 里的代码访问不到。
**应用**:必须动上游文件时,遵守两条:①能追加在文件末尾的绝不插在中间(上游改中间,我们改末尾,几乎不冲突);②每处改动都带 `[翻译]` 注释标记,方便 ⌘F 盘点。验证手段:`git diff --stat` 应该只见 `+` 几乎不见 `-`。

## L6 · Storyboard 是 XML,能不碰就不碰

**现象**:工具栏按钮实体定义在 `Main.storyboard` 里,直觉上加按钮要改它。
**应用**:上游自己就有"纯代码创建按钮"的先例(ArticleExtractorButton),照抄即可。设置页新行同理 —— 上游在代码里动态增删行(`accounts.count + 1`),跟着这个模式走,932 行的 Settings.storyboard 一个字符都没碰。

## L7 · LLM 的输出永远不能直接信

**现象**(全部真实发生过):把译文包在 ``` 里;把原文和译文一起吐出来(中英对照);整段原样返回不翻译。
**教训**:**提示词是软约束,代码才是硬的。** 提示词里明令禁止的事,模型照样会干。
**应用**:三层防线:①提示词明令禁止;②代码清洗(剥代码围栏、切掉回显的原文);③事后自检(JS 本地判断哪些组还是英文/混了原文,重翻)。每层都要保守设计,宁可漏抓不可误杀。

## L8 · "第一块翻完就点亮按钮"是错误的产品判断

**现象**:我以为提前点亮按钮能传达"已经开始了",实际用户的感受是"按钮说完成了内容却还在变,我不知道该不该等"。
**教训**:**状态指示器的语义必须诚实**:转圈=还在干活,实心=真的干完了。"内容在逐段变中文"本身已经是进行中的信号,不需要按钮抢戏。
**应用**:任何异步流程,完成态只在**全部**结束后设置。

## L9 · 均分的并行块对"顺序阅读"是错的

**现象**:其余内容均分 4 组,每组几千字符 → 用户读完先导块,要干等第 1 组(和最后一组一样大、一样慢),体验断粮。
**教训**:优化目标错了 —— 均分优化的是**总完成时间**,但用户按顺序读,真正该优化的是**第 N 块在被读到之前就绪**。
**应用**:渐进式分组:第 1 组最小(1000 字符),逐组翻倍,4000 封顶。供给节奏跟着阅读节奏走。

## L10 · 标题在正文容器外面

**现象**:只切 `#bodyContainer` 里的内容,标题(`.articleTitle`)漏翻了。
**应用**:改任何"处理正文"的逻辑时,记得问一句:**标题、日期、作者这些容器外的东西要不要一起处理?** 目前只处理标题,日期作者有意不翻。

## L11 · 游离的 Task 不会跟着父任务取消

**现象**:标题翻译用了独立的 `Task {}`,父任务取消时它还活着 —— 用户翻页后,它可能把上一篇的标题译文安到下一篇文章头上(WebViewController 是复用的)。
**应用**:凡是"发射后不管"的 Task:①保存句柄;②`defer { task?.cancel() }`;③任务内部在写回 UI 前检查 `Task.isCancelled`。三样都要。

## L12 · 主题会改 DOM 结构,选择器必须带兜底链

**现象**:8 个内置主题里,Biblioteca 的正文容器是 `body-container` 而不是 `bodyContainer`。只认一个 id 的话,用户换主题后翻译按钮静默失灵。
**应用**:`translation.js` 里所有选择器都是候选链(id → 变体 id → class → 变体 class → 标签名),全落空才报"主题不受支持"。**永远不要静默失败。**

## L13 · Shared/ 的代码会被 macOS 一起编译

**现象**:`TranslationController` 用了 UIKit,iOS 编译过了,macOS 版会挂 —— 而且平时只跑 iOS 根本发现不了。
**应用**:①UIKit 相关文件整个包 `#if os(iOS)`;②**每次改 Shared/ 后必须跑一次 macOS 免签名编译**(命令见 NOTES-progress 第六节)。macOS 常规编译会卡在签名,要加 `CODE_SIGNING_ALLOWED=NO` 系列参数。

## L14 · UIPageViewController 的翻页回调只覆盖手势

**现象**:按钮状态重置挂在 `didFinishAnimating` 上,结果只有手指滑动翻页会触发;右上角箭头、底部"下一篇未读"、列表点选都不走这个回调。
**教训**:修 bug 时别只堵报上来的那个口子 —— 要找**所有路径的必经之地**。
**应用**:挂在 `article` 属性的 `didSet` 上(一切换文章方式最终都经过它)。第二层保险:按钮点击时不信 Swift 记的状态,先问网页当前真实显示的是什么。

## L16 · "替换失败"被静默吞掉,表象是"某组总是最后才翻好"

**现象**:用户报告分块并行时第 1 组总是压轴才变中文,后面的组都好了它还是英文。
排查发现两个叠加机制:
① 组译文替换回页面(applyGroup)失败时,代码写的是 `_ = try? ...` —— 结果被扔掉,
不重试、不计数。这一组一直显示原文,**拖到最后的自检阶段才被救回来**,所以永远最后。
② 引用型博客(Michael Tsai / Daring Fireball,恰是用户的主要订阅)常有一整段
几千字符的 `<blockquote>`,顶层切分切不动它,谁分到它谁就巨慢。
**教训**:异步流水线里**每一步的失败都必须有去向**(重试/计数/上报)。
`_ = try?` 等于把故障扫进地毯下,最后以离奇的表象冒出来。
**应用**:①替换结果并入重试循环;②超大元素下钻一层按子元素切
(同一组绝不跨父节点,否则替换会插错位置);③模型返回裸文本时用原标签包回;
④加了每组"大小/耗时/结果"日志 —— **下次再有性能问题,先看 `[翻译]` 日志,
用数据定位,不猜**。

## L17 · 改提示词必须让旧缓存作废

**现象**:加了"专有名词保留英文"的新规则,但已缓存文章命中缓存直接秒开 ——
旧译文是按旧规则翻的,新规则永远轮不到生效,用户会以为改了没用。
**应用**:缓存键里加了 `promptGeneration` 版本号(TranslationCache.swift),
**每次大改提示词都要把它 +1**,全部旧缓存自动作废重翻。

## L18 · 缓存指纹不能取自"活的"DOM HTML

**现象**:用户报告缓存"时灵时不灵" —— 同一篇翻过的文章,有时秒开,有时明显重新请求。
**原因**:判断"原文变没变"的指纹取的是正文 innerHTML 的哈希。但页面加载后,
NetNewsWire 自带脚本会**异步**改 HTML(图片查看器装饰、懒加载等),
图片加载到哪张取决于用户点按钮的时机 → 同一篇文章两次取到的 HTML 不同 →
哈希不同 → 被误判为"文章更新了" → 重新翻译。
**教训**:**指纹必须取自稳定的东西。** "时灵时不灵"类 bug 十有八九是
输入里混进了时序相关的变量。
**应用**:指纹改为正文**纯文字**(normalizeSpace(textContent))的哈希 ——
文字才是我们翻译的对象,也不受图片装饰影响。同时缓存版本号 +1 清掉坏指纹的旧条目。

## L19 · customView 的工具栏按钮必须有尺寸约束,否则会被压成 0 宽永久消失

**现象**(2026-07-21 用户报告):文章页底部工具栏的"阅读视图"和"翻译"两个按钮同时消失;
一旦出现,之后**所有**文章都没有这两个按钮,重启 app 才恢复。
用户复现方式:反复进文章 → 退回列表 → 再进文章。

**定位过程(值得复用的方法)**:
1. 抓模拟器日志:`xcrun simctl spawn booted log show --predicate 'process == "NetNewsWire"'`
2. 关键证据:`Unable to simultaneously satisfy constraints` 里出现
   `'UIView-Encapsulated-Layout-Width' ButtonWrapper.width == 0` —— 宽度被算成 0
3. 对照消失的是哪些按钮:○☆⌄⬆️(标准 bar button)全在,
   📄阅读视图 和 💬翻译(**两个 customView 按钮**)全没 —— 一个不多一个不少

**原因**:两个按钮都只设了 `frame = 44×44`、没有任何尺寸约束,
宽度靠"图标撑出来的固有尺寸"。而两者在转圈状态都会 `setImage(nil)` ——
图标一没,固有尺寸变 0,iOS 26 的 Liquid Glass 工具栏把它算成 0 宽并**永久塌陷**
(ArticleViewController 是常驻实例,工具栏不重建,所以之后所有文章都受影响)。

⚠️ 阅读视图按钮是**上游自己的代码**,同样的毛病;我照抄它的写法等于复制了隐患。

**修复**:给两个 customView 都加死 44×44 约束
(`translatesAutoresizingMaskIntoConstraints = false` + width/height anchor)。
上游那个按钮的约束加在本 fork 的 `installTranslationButton()` 里,不动上游原有代码行。

**验证(受控对比)**:修复前日志里 `ButtonWrapper ... width == 0` 出现 **40 次**
(其中最早两次在启动过程中、用户尚未交互时);修复后同样的启动路径 **0 次**。

**教训**:
- **把 UIButton 当 `UIBarButtonItem(customView:)` 用时,一定要给显式尺寸约束**,
  永远不要依赖"图标撑出来的固有尺寸" —— 任何会把图标置空的状态都会让它塌掉
- "一旦发生就再也不恢复"这类 bug,几乎都指向**常驻对象上的一次性布局塌陷**,
  找那个不会被重建的东西
- 症状里"哪些坏了、哪些没坏"的**边界**是最强的线索:
  这次"只有两个 customView 按钮消失"直接锁定了原因

## L15 · 上游 CLAUDE.md 与我们的规则文件重名

**现象**:上游仓库自带英文版 `CLAUDE.md`,clone 后覆盖了用户准备的中文规则。
**应用**:上游版已改存为 `CLAUDE.upstream.md`(git 不跟踪)。将来 `git pull upstream` 如果 CLAUDE.md 冲突,**保留我们的中文版**,把上游新版本另存覆盖 `CLAUDE.upstream.md`。
