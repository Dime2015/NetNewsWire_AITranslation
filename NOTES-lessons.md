## L1 · 全新 clone 后第一次编译必失败

**现象**:`cannot find 'SecretKey' in scope`,两处报错都在 Feedly 相关文件。
**原因**:`SecretKey.swift` 被 .gitignore 排除,由编译期脚本生成,但生成时机晚于 Secrets 模块编译。先有鸡还是先有蛋,只在第一次出现。
**应用**:再编译一次就好。**绝对不要去"修"那两个报错的文件** —— 那是 A 级禁区,而且根本没坏。

## L2 · 仓库根目录的新文件夹不参与编译

**现象**:原计划把新代码放根目录 `Translation/`,但那里的代码 Xcode 根本看不见。
**原因**:本工程用 Xcode 16 的文件系统同步组机制,只有 `Mac/ iOS/ Shared/ Modules/ Widget/ Tests/ Technotes/ xcconfig/` 这 8 个文件夹里的新文件会自动进编译,根目录不在其中。
**应用**:所有新代码放 `Shared/Translation/`。**新增文件前先确认目标文件夹在同步组清单里**(grep pbxproj 搜 `PBXFileSystemSynchronizedRootGroup`)。

## L3 · 用探针文件实测,不要只靠推理

**现象**:「`Shared/Translation/` 下的 .js/.json 会不会被打进 app 包?」—— 光看工程文件推不出来。
**应用**:放一个一行的探针文件 → 编译 → 直接 `ls` 构建产物的 app 包看它在不在 → 删探针。**一次真实构建胜过十次推理**,本项目用这招验证过 3 次关键假设。

## L4 · NSLocalizedString 会污染上游的共享文件

**现象**:代码里用了 `NSLocalizedString`,编译后发现 `Shared/Localizable.xcstrings`(上游共用大文件)被 Xcode 自动塞进了我们的中文文案。
**教训**:这种"编译器自动改文件"的副作用,肉眼看代码根本发现不了,是 merge 冲突的隐形炸弹。
**应用**:本 fork **禁用 NSLocalizedString**,直接写中文字符串(单用户 app 不需要多语言)。每次编译后跑 `git status` 检查有没有意料之外的文件被改。

## L5 · 上游文件的 private 成员够不着 → 用"纯末尾追加"的扩展

**现象**:`webView`、`currentWebViewController`、`toolbarItems` 都是 private/内部的,`Shared/Translation/` 里的代码访问不到。
**应用**:必须动上游文件时,遵守两条:①能追加在文件末尾的绝不插在中间(上游改中间,我们改末尾,几乎不冲突);②每处改动都带 `[翻译]` 注释标记,方便 ⌘F 盘点。验证手段:`git diff --stat` 应该只见 `+` 几乎不见 `-`。

## L6 · Storyboard 是 XML,能不碰就不碰

**现象**:工具栏按钮实体定义在 `Main.storyboard` 里,直觉上加按钮要改它。
**应用**:上游自己就有"纯代码创建按钮"的先例(ArticleExtractorButton),照抄即可。设置页新行同理 —— 上游在代码里动态增删行(`accounts.count + 1`),跟着这个模式走,932 行的 Settings.storyboard 一个字符都没碰。

## L7 · LLM 的输出永远不能直接信

**现象**(全部真实发生过):把译文包在 ``` 里;把原文和译文一起吐出来(中英对照);整段原样返回不翻译。
**教训**:**提示词是软约束,代码才是硬的。** 提示词里明令禁止的事,模型照样会干。
**应用**:三层防线:①提示词明令禁止;②代码清洗(剥代码围栏、切掉回显的原文);③事后自检(JS 本地判断哪些组还是英文/混了原文,重翻)。每层都要保守设计,宁可漏抓不可误杀。

## L8 · "第一块翻完就点亮按钮"是错误的产品判断

**现象**:我以为提前点亮按钮能传达"已经开始了",实际用户的感受是"按钮说完成了内容却还在变,我不知道该不该等"。
**教训**:**状态指示器的语义必须诚实**:转圈=还在干活,实心=真的干完了。"内容在逐段变中文"本身已经是进行中的信号,不需要按钮抢戏。
**应用**:任何异步流程,完成态只在**全部**结束后设置。

## L9 · 均分的并行块对"顺序阅读"是错的

**现象**:其余内容均分 4 组,每组几千字符 → 用户读完先导块,要干等第 1 组(和最后一组一样大、一样慢),体验断粮。
**教训**:优化目标错了 —— 均分优化的是**总完成时间**,但用户按顺序读,真正该优化的是**第 N 块在被读到之前就绪**。
**应用**:渐进式分组:第 1 组最小(1000 字符),逐组翻倍,4000 封顶。供给节奏跟着阅读节奏走。

## L10 · 标题在正文容器外面

**现象**:只切 `#bodyContainer` 里的内容,标题(`.articleTitle`)漏翻了。
**应用**:改任何"处理正文"的逻辑时,记得问一句:**标题、日期、作者这些容器外的东西要不要一起处理?** 目前只处理标题,日期作者有意不翻。

## L11 · 游离的 Task 不会跟着父任务取消

**现象**:标题翻译用了独立的 `Task {}`,父任务取消时它还活着 —— 用户翻页后,它可能把上一篇的标题译文安到下一篇文章头上(WebViewController 是复用的)。
**应用**:凡是"发射后不管"的 Task:①保存句柄;②`defer { task?.cancel() }`;③任务内部在写回 UI 前检查 `Task.isCancelled`。三样都要。

## L12 · 主题会改 DOM 结构,选择器必须带兜底链

**现象**:8 个内置主题里,Biblioteca 的正文容器是 `body-container` 而不是 `bodyContainer`。只认一个 id 的话,用户换主题后翻译按钮静默失灵。
**应用**:`translation.js` 里所有选择器都是候选链(id → 变体 id → class → 变体 class → 标签名),全落空才报"主题不受支持"。**永远不要静默失败。**

## L13 · Shared/ 的代码会被 macOS 一起编译

**现象**:`TranslationController` 用了 UIKit,iOS 编译过了,macOS 版会挂 —— 而且平时只跑 iOS 根本发现不了。
**应用**:①UIKit 相关文件整个包 `#if os(iOS)`;②**每次改 Shared/ 后必须跑一次 macOS 免签名编译**(命令见 NOTES-progress 第六节)。macOS 常规编译会卡在签名,要加 `CODE_SIGNING_ALLOWED=NO` 系列参数。

## L14 · UIPageViewController 的翻页回调只覆盖手势

**现象**:按钮状态重置挂在 `didFinishAnimating` 上,结果只有手指滑动翻页会触发;右上角箭头、底部"下一篇未读"、列表点选都不走这个回调。
**教训**:修 bug 时别只堵报上来的那个口子 —— 要找**所有路径的必经之地**。
**应用**:挂在 `article` 属性的 `didSet` 上(一切换文章方式最终都经过它)。第二层保险:按钮点击时不信 Swift 记的状态,先问网页当前真实显示的是什么。

## L16 · "替换失败"被静默吞掉,表象是"某组总是最后才翻好"

**现象**:用户报告分块并行时第 1 组总是压轴才变中文,后面的组都好了它还是英文。
排查发现两个叠加机制:
① 组译文替换回页面(applyGroup)失败时,代码写的是 `_ = try? ...` —— 结果被扔掉,
不重试、不计数。这一组一直显示原文,**拖到最后的自检阶段才被救回来**,所以永远最后。
② 引用型博客(Michael Tsai / Daring Fireball,恰是用户的主要订阅)常有一整段
几千字符的 `<blockquote>`,顶层切分切不动它,谁分到它谁就巨慢。
**教训**:异步流水线里**每一步的失败都必须有去向**(重试/计数/上报)。
`_ = try?` 等于把故障扫进地毯下,最后以离奇的表象冒出来。
**应用**:①替换结果并入重试循环;②超大元素下钻一层按子元素切
(同一组绝不跨父节点,否则替换会插错位置);③模型返回裸文本时用原标签包回;
④加了每组"大小/耗时/结果"日志 —— **下次再有性能问题,先看 `[翻译]` 日志,
用数据定位,不猜**。

## L17 · 改提示词必须让旧缓存作废

**现象**:加了"专有名词保留英文"的新规则,但已缓存文章命中缓存直接秒开 ——
旧译文是按旧规则翻的,新规则永远轮不到生效,用户会以为改了没用。
**应用**:缓存键里加了 `promptGeneration` 版本号(TranslationCache.swift),
**每次大改提示词都要把它 +1**,全部旧缓存自动作废重翻。

## L18 · 缓存指纹不能取自"活的"DOM HTML

**现象**:用户报告缓存"时灵时不灵" —— 同一篇翻过的文章,有时秒开,有时明显重新请求。
**原因**:判断"原文变没变"的指纹取的是正文 innerHTML 的哈希。但页面加载后,
NetNewsWire 自带脚本会**异步**改 HTML(图片查看器装饰、懒加载等),
图片加载到哪张取决于用户点按钮的时机 → 同一篇文章两次取到的 HTML 不同 →
哈希不同 → 被误判为"文章更新了" → 重新翻译。
**教训**:**指纹必须取自稳定的东西。** "时灵时不灵"类 bug 十有八九是
输入里混进了时序相关的变量。
**应用**:指纹改为正文**纯文字**(normalizeSpace(textContent))的哈希 ——
文字才是我们翻译的对象,也不受图片装饰影响。同时缓存版本号 +1 清掉坏指纹的旧条目。

## L19 · customView 的工具栏按钮必须有尺寸约束,否则会被压成 0 宽永久消失

**现象**(2026-07-21 用户报告):文章页底部工具栏的"阅读视图"和"翻译"两个按钮同时消失;
一旦出现,之后**所有**文章都没有这两个按钮,重启 app 才恢复。
用户复现方式:反复进文章 → 退回列表 → 再进文章。

**定位过程(值得复用的方法)**:
1. 抓模拟器日志:`xcrun simctl spawn booted log show --predicate 'process == "NetNewsWire"'`
2. 关键证据:`Unable to simultaneously satisfy constraints` 里出现
   `'UIView-Encapsulated-Layout-Width' ButtonWrapper.width == 0` —— 宽度被算成 0
3. 对照消失的是哪些按钮:○☆⌄⬆️(标准 bar button)全在,
   📄阅读视图 和 💬翻译(**两个 customView 按钮**)全没 —— 一个不多一个不少

**原因**:两个按钮都只设了 `frame = 44×44`、没有任何尺寸约束,
宽度靠"图标撑出来的固有尺寸"。而两者在转圈状态都会 `setImage(nil)` ——
图标一没,固有尺寸变 0,iOS 26 的 Liquid Glass 工具栏把它算成 0 宽并**永久塌陷**
(ArticleViewController 是常驻实例,工具栏不重建,所以之后所有文章都受影响)。

⚠️ 阅读视图按钮是**上游自己的代码**,同样的毛病;我照抄它的写法等于复制了隐患。

**修复**:给两个 customView 都加死 44×44 约束
(`translatesAutoresizingMaskIntoConstraints = false` + width/height anchor)。
上游那个按钮的约束加在本 fork 的 `installTranslationButton()` 里,不动上游原有代码行。

**验证(受控对比)**:修复前日志里 `ButtonWrapper ... width == 0` 出现 **40 次**
(其中最早两次在启动过程中、用户尚未交互时);修复后同样的启动路径 **0 次**。

**教训**:
- **把 UIButton 当 `UIBarButtonItem(customView:)` 用时,一定要给显式尺寸约束**,
  永远不要依赖"图标撑出来的固有尺寸" —— 任何会把图标置空的状态都会让它塌掉
- "一旦发生就再也不恢复"这类 bug,几乎都指向**常驻对象上的一次性布局塌陷**,
  找那个不会被重建的东西
- 症状里"哪些坏了、哪些没坏"的**边界**是最强的线索:
  这次"只有两个 customView 按钮消失"直接锁定了原因

## L20 · 本地化:翻译只能写进上游的 .xcstrings,但可以做到"只增行"

**背景**(2026-07-21):把界面完全汉化,需要处理 436 条字符串。

**试过但失败的路**:"完全不碰上游文件,另起一份自己的中文表"。
Xcode 直接拒绝:`Localizable.xcstrings cannot co-exist with other .strings tables with the same name`。
**字符串目录和同名 .strings 不能并存,翻译只能写进上游那个目录。**

**关键手法 —— 复刻 Xcode 的 JSON 格式**:
Xcode 写 .xcstrings 用的格式很特殊(键与冒号间有空格、空对象带空行、**文件末尾没有换行**)。
用普通 `json.dump` 会把整个文件重排,`git diff` 变成"整份文件都改了",
`git pull upstream` 时冲突根本没法读。
`i18n/inject.py` 复刻了该格式,并在写入前自检"能否字节级还原原文件",不能就拒写。
结果:注入 436 条后 diff 只有新增行,逐 key 校验确认原有内容一字节未变。

**教训**:
- **要改一个由工具生成的文件时,先做"读出来再写回去,是否字节一致"的往返测试。**
  一致才动手。这个测试 10 分钟,能省掉后面无数次无法阅读的 merge 冲突。
- 末尾换行这种细节最容易被忽略 —— 往返测试会直接抓出来。
- 注入脚本要**双向报告**:哪些 key 没翻、翻译表里哪些 key 在目录里找不到。
  后者能立刻抓出弯引号/不间断空格抄错,不会静默失败。(实测 436 条零抄错)

**衍生发现**:
- Storyboard 分两类:在 `Base.lproj/` 里的可以纯新增 `<语言>.lproj/xxx.strings` 翻译(零上游改动);
  不在任何 `.lproj` 里的(本项目的 Settings/Account/Add/Inspector)必须先移动文件才能翻。
  判断方法:编译后看它落在 app 包根目录还是 `Base.lproj/` 里。
- 语言选择器的可选项要从 `Bundle.main.localizations` 动态读,**不要写死语言列表** ——
  以后加日语时选择器自动多一行,一行代码都不用改。
- 编译时 Xcode 会自动往 `project.pbxproj` 的 `knownRegions` 加一行新语言,这是必要的,不要还原。

完整操作手册见 `NOTES-i18n.md`。

## L21 · .xcstrings 的"设备变体":译文结构必须和英文一致

**现象**:界面上本该显示"On My iPhone"的地方,汉化后变成了字面的 `我的%@` —— 占位符没被替换。
**原因**:该字符串英文侧用的是 **`variations.device`** 结构
(iphone → "On My iPhone"、ipad → "On My iPad"、mac → "On My Mac"…),
**根本没有 %@ 这个占位符**。我按普通字符串翻成 `"我的 %@"`,自然没有东西可替换。
**教训**:翻译前**先看英文侧那一条是什么结构**。.xcstrings 里除了普通 `stringUnit`,
还有 `variations`(按设备 device / 按复数 plural)。译文必须用同样的结构,
否则不是显示错乱就是原样吐出占位符。
**应用**:`i18n/inject.py` 现在支持在翻译表里把值写成字典,
自动生成 `variations.device` 结构。检查是否还有其它变体条目:
`grep -l '"variations"' **/*.xcstrings`

## L22 · 改上游的界面前,先把「可调的数值」搬进自己的文件

**背景**(2026-07-21):用户要求改文章列表的样子。考古发现列表**不是 storyboard、
不是 Auto Layout**,而是逐帧手算坐标的 manual layout —— 字号、间距、颜色全是
`MainTimelineCellLayout.swift` / `MainTimelineCell.swift` 里的常量。这两个都是上游文件。

**现象(如果不管)**:每调一次字号就在上游文件里改一个数字。改十次就在上游文件里
留下十处散落的改动,`git pull upstream` 时每一处都是潜在冲突,而且冲突长得像
「上游把 12 改成 14,你把 12 改成 16」这种最难判断的样子。

**教训**:**改上游界面代码时,真正该做的第一件事不是改样子,是把「会反复变的数值」
搬到自己的文件里。** 上游文件的 diff 从此永远停在「一行换一行的引用」,
以后无论调多少次外观,上游文件都不再变。

**应用**:新建 `iOS/MainTimeline/TimelineStyle.swift`(本 fork 新增,上游不存在),
把常量按**原值**搬进去;上游文件里 `static let cellPadding = UIEdgeInsets(...)`
改成 `static let cellPadding = TimelineStyle.cellPadding`,每处带 `// [界面]` 标记。
规则已写进 CLAUDE.md 第 2 节「D 级 · 界面改造专用」。

**这一步的验收标准是「界面零变化」,而且必须真的去量**:
搬运时抄错一个数字,肉眼根本看不出来。做法是找一个**内容固定**的界面做对照 ——
本项目用的是「设置 → 文章列表布局」那两条预览(内容永远是 "At The Sign Of The
Prancing Pony"),改动前后各截一张图,去掉状态栏(时钟会变)后逐像素比。
**实测 280 万像素 0 处不同**,才敢说搬对了。
拿真实文章列表比是不行的 —— 已读状态、时间都在变,差异全是噪音。

## L23 · 正文页的样式可以「追加一层」,不必改上游的样式表

**背景**(2026-07-21):想改正文阅读页的排版。最直觉的做法是改
`Shared/Article Rendering/stylesheet.css` —— 但那是上游高频改动的文件。

**发现的通道**:`WebViewConfiguration.swift` 里有一个脚本清单
`let filenames = ["main", "main_ios", "newsfoot"]`,这些 .js 会在**每次页面加载前**
自动注入。往这个数组里加一个我们自己的脚本名(**上游改动 = 一行里加一个单词**),
脚本在页面加载完成后往 `<head>` 追加一个 `<style>`,就得到了一层我们完全控制的样式。
上游的 CSS 一个字都不用动,而且对 8 套主题同时生效。

**三个必须知道的细节(都是实测出来的,不是推的)**:

1. **样式必须插进 `<head>`,不能插进 `<body>`。**
   上游 `main.js` 的 `stripStyles()` 会把 `<body>` 里所有 `<style>` 全部删掉。
2. **同优先级时后加的赢,不需要 `!important`。**
   实测:我们的 `.articleTitle a:link { color: #cc0000 }` 盖过了上游同名同优先级的规则。
   这是整条通道的地基,所以专门用探针验证过,没有凭"CSS 应该是这样"下结论。
3. **标题颜色写在 `.articleTitle a:link` 上,不在 `.articleTitle h1` 上。**
   第一次探针写的是 `h1`,颜色没变 —— 因为标题文字被包在 `<a>` 里,
   子元素上的规则赢了父元素。**"没生效"不一定是通道坏了,也可能是选择器选错了对象。**

**方法本身值得复用(就是 L3 的探针法)**:先塞一条**故意难看**的样式
(正文底色刷成琥珀黄),截图确认真的变了,再清空。
如果一开始就写空 CSS,那么"界面没变化"既可能说明链路是通的,
也可能说明链路根本没接上 —— **两种情况长得一模一样,等于没验证。**

**衍生发现**:iOS 版**没有**正文字号滑块(那是 macOS 专属,
`ArticleRenderer.swift` 里 `text_size_class` 包在 `#if os(macOS)` 里)。
iOS 的正文字号来自系统的动态字体设置,由 `styleSubstitutions()` 注入成 CSS 的 `font-size`。
用户想调正文字号,要么改系统设置,要么由我们在这层 CSS 里覆盖。

## L24 · 换 app 图标:别覆盖上游的 PNG,改一行 xcconfig 指到自己的图标集

**背景**(2026-07-21):用户提供了一张新的 app 图标。

**最直觉但错误的做法**:直接覆盖
`iOS/Resources/Assets.xcassets/AppIcon.appiconset/` 里上游的三个 PNG。
后果:以后上游一换图标就是**二进制冲突** —— git 没法帮你合二进制文件,
只能手工挑一个,而且 `git diff` 完全看不出发生了什么。

**正确做法**:图标集的名字是一个**文本设置**,在
`xcconfig/common/NetNewsWire_ios_target_common.xcconfig` 里:
`ASSETCATALOG_COMPILER_APPICON_NAME = AppIcon`。
所以另建一套 `AppIconCustom.appiconset`,把设置指过去就行,上游那套原封不动。

⚠️ **但不要改那个 common 文件** —— 小组件扩展和 Siri 扩展也 include 它,
它们各自的图标资源里没有 `AppIconCustom`。
应该改在**主 app 自己的** `xcconfig/NetNewsWire_iOSapp_target.xcconfig` 里追加一行:
因为它写在 `#include` 之后,会覆盖掉继承来的值,而且只影响主 app。

**核实手段**(不要凭"应该生效了"):
```bash
/usr/libexec/PlistBuddy -c "Print :CFBundleIcons" <app包>/Info.plist
# 应该看到 CFBundleIconName = AppIconCustom
xcrun assetutil --info <app包>/Assets.car    # 看实际编进去了哪几种外观
```

## L26 · app 图标素材必须是「满铺方图」,自带圆角会变成框里套框

**现象**(2026-07-21 用户指出):第一版图标素材本身就是一张**带圆角边框**的图。
直接拿去当 app 图标,iOS 又套了一层自己的圆角遮罩 —— 结果是**框里套框**,
整个图案往里缩了一圈,视觉上明显比旁边的系统图标"小一号",很难看。

**我当时漏掉了它,值得记下来为什么**:
我确实检查了,但我检查的问题是「**四角有没有毛刺**」——
我算了一下 iOS 的遮罩半径比原图自带的圆角切得更多,所以推断白色的角会被完全裁掉,
然后放大截图确认了角上确实干净,就下了"没问题"的结论。
**这个推理和这次观察都没错,错在问题问窄了。** 我核对的是"边缘有没有瑕疵",
而真正的毛病是"整个图案的占比不对" —— 一个只看角、不看整体的检查,永远发现不了它。

**教训**:
- **app 图标素材必须是内容满铺到边缘的正方形图**,不要自带圆角、不要留白边、
  不要自带投影。圆角由 iOS 统一加,加两次就是错。
- 更一般的:**验收视觉改动时,不要只盯着自己预判的那个风险点。**
  先整体看一眼"它和周围的东西放在一起协调吗",再去抠细节。
  我这次是先抠了细节,而整体那一眼从头到尾没看。

**顺带发现(第二版素材实测)**:满铺之后,靠近四角的内容会被 iOS 的圆角**真的切掉**。
第二版图右下角的 wifi 图形最外圈那道弧就被切断了。
所以"满铺"不等于"可以画到边":**重要元素要离边缘留出足够距离**
(iOS 的圆角半径约为边长的 22%,四角那一块基本都会没)。
验收方法:装到模拟器 → 主屏截图 → 把图标那一小块裁出来放大 6 倍看。
放大之前在正常尺寸下是看不出来的。

**实测出的经验数值(第三版通过)**:右下角的元素**离下边缘 8.2%、离右边缘 10.5%** 时
完整不被切;第二版是 5.1% / 9.0%,被切了。
所以角落元素的安全边距至少留 **8%** 以上。

⚠️ **这个问题只能在素材里解决,处理图片时避不开** ——
我们这边只是把 1254 等比缩到 1024,切图案的是 iOS 显示时套的圆角遮罩。

## L25 · iOS 26 的图标有三种外观,「色调」那种是系统按亮度生成的

**现象**:换完图标,浅色、深色主屏都好看;把主屏切到**「色调」**外观时,
别的图标都变成蓝色单色,**我们的还是白底黑字**,像块白板贴在一排蓝图标里。

**排查**:`xcrun assetutil --info Assets.car` 显示编进去的只有「默认」和「深色」两份,
**单色版根本不是我们提供的,是 iOS 自己生成的** ——
它把图转灰度,再把灰度值映射到用户选的色调上。

**原因**:我们这张是**浅底**图(白报纸底 + 黑 R),灰度后几乎全白,
映射后停在色阶最亮那端 —— 看起来就是"没上色"。
试过把 tinted 那条从 Contents.json 删掉让系统自己来,结果一模一样,
因为系统的输入还是这张浅底图。

**后续(2026-07-21 已解决)**:用户提供了一张**黑底白 R 的灰度图**作为 mono 版,
在 Contents.json 里显式登记成 `luminosity: tinted`。
「色调」外观下立刻变成深底浅字,和系统图标完全一致。**印证了下面这条教训。**

另有一个验证时的坑:改完深色素材后用 `xcrun simctl ui booted appearance dark`
切系统深色模式,图标**没变** —— 不是素材没生效,而是
**主屏图标外观是独立设置**,当时被钉在「默认」上。
测深色图标必须走 主屏长按 → 编辑 → 自定 → 深色。

**教训**:
- **单色图标的观感由原图的明暗分布决定,不由配置决定。**
  想在「色调」模式下好看,素材本身必须是**深底浅字**的高对比灰度图。
- 更一般的:凡是"系统会替你自动生成一个变体"的机制(单色图标、深色模式颜色、
  SF Symbol 的各种渲染模式),**生成质量取决于你喂进去的素材,不取决于你的设置**。
  设置对了但结果难看,先怀疑素材。
- 验收一个图标必须**三种外观都看**:长按主屏 → 编辑 → 自定 → 默认/深色/透明/色调。
  只看默认那种会漏掉这个问题。

## L27 · 对照物只有在"两种假设会给出不同结果"时才算证据

**这条是我犯了错之后才写对的,过程本身就是教训。**

**现象**(2026-07-21):用户问,真机上 app 图标会跟系统深浅色实时变,
模拟器里为什么不变 —— 是模拟器的问题还是代码没写好?

**我当时的推理**:
1. `assetutil` 确认三种外观都编进去了 ✅
2. 手动选「深色」「色调」,我们的素材都能正确显示 ✅
3. **"决定性证据":Apple 自己的「通讯录」「文件」在深色模式下也是浅色的** ——
   别人家也不跟随,所以不是我们的锅
4. 重启 SpringBoard 后仍不变(排除了"没刷新")

→ 结论:**模拟器不支持自动切换**。

**这个结论是错的。** 用户随后指出:主屏「自定」面板里选「深色」之后,
**下面还有一排子选项「始终 / 自动」**,要选「自动」才会跟随系统。
按用户说的一试,立刻正常 —— 我们的素材一点问题都没有。

**错在哪:第 3 条根本不是证据。**
那条我自以为"决定性"的对照,在两种假设下**预测完全相同**:

| | 假设 A:模拟器不支持 | 假设 B:全局开关没开 |
|---|---|---|
| Apple 的图标会不会跟随? | 不会 | **也不会** |

因为那个开关(当时停在「默认」)作用于**所有** app,Apple 自己的也不例外。
一条对两种假设给出同样预测的观察,**信息量是零**,
我却把它当成了压倒性证据,还因为它"感觉很硬"而停止了继续找原因。

**教训**:
- **对照物有没有用,取决于"两种假设会不会给出不同结果"。**
  动手前先问一句:如果是 A,我会看到什么?如果是 B 呢?
  **两个答案一样,这个实验就别做了,做了也白做。**
- 特别当心**全局性的设置/开关** —— 它天然会让"我的 app"和"别人的 app"
  表现一致,从而伪装成"环境的锅"。凡是"所有 app 都这样"的现象,
  在断定是平台问题之前,先找一遍有没有一个管着所有 app 的开关。
- **"我已经有决定性证据了"这个念头本身就是危险信号。**
  这次我正是因为觉得第 3 条够硬,就没再去翻那个面板的下一层。
- 仍然成立的一条:排查时要分清**"没生效"还是"生效了没刷新"**,
  两者表象一样、修法不同(这次用重启 SpringBoard 排除了后者,这步是对的)。
