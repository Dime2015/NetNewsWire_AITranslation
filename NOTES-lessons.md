## L1 · 全新 clone 后第一次编译必失败

**现象**:`cannot find 'SecretKey' in scope`,两处报错都在 Feedly 相关文件。
**原因**:`SecretKey.swift` 被 .gitignore 排除,由编译期脚本生成,但生成时机晚于 Secrets 模块编译。先有鸡还是先有蛋,只在第一次出现。
**应用**:再编译一次就好。**绝对不要去"修"那两个报错的文件** —— 那是 A 级禁区,而且根本没坏。

## L2 · 仓库根目录的新文件夹不参与编译

**现象**:原计划把新代码放根目录 `Translation/`,但那里的代码 Xcode 根本看不见。
**原因**:本工程用 Xcode 16 的文件系统同步组机制,只有 `Mac/ iOS/ Shared/ Modules/ Widget/ Tests/ Technotes/ xcconfig/` 这 8 个文件夹里的新文件会自动进编译,根目录不在其中。
**应用**:所有新代码放 `Shared/Translation/`。**新增文件前先确认目标文件夹在同步组清单里**(grep pbxproj 搜 `PBXFileSystemSynchronizedRootGroup`)。

## L3 · 用探针文件实测,不要只靠推理

**现象**:「`Shared/Translation/` 下的 .js/.json 会不会被打进 app 包?」—— 光看工程文件推不出来。
**应用**:放一个一行的探针文件 → 编译 → 直接 `ls` 构建产物的 app 包看它在不在 → 删探针。**一次真实构建胜过十次推理**,本项目用这招验证过 3 次关键假设。

## L4 · NSLocalizedString 会污染上游的共享文件

**现象**:代码里用了 `NSLocalizedString`,编译后发现 `Shared/Localizable.xcstrings`(上游共用大文件)被 Xcode 自动塞进了我们的中文文案。
**教训**:这种"编译器自动改文件"的副作用,肉眼看代码根本发现不了,是 merge 冲突的隐形炸弹。
**应用**:本 fork **禁用 NSLocalizedString**,直接写中文字符串(单用户 app 不需要多语言)。每次编译后跑 `git status` 检查有没有意料之外的文件被改。

## L5 · 上游文件的 private 成员够不着 → 用"纯末尾追加"的扩展

**现象**:`webView`、`currentWebViewController`、`toolbarItems` 都是 private/内部的,`Shared/Translation/` 里的代码访问不到。
**应用**:必须动上游文件时,遵守两条:①能追加在文件末尾的绝不插在中间(上游改中间,我们改末尾,几乎不冲突);②每处改动都带 `[翻译]` 注释标记,方便 ⌘F 盘点。验证手段:`git diff --stat` 应该只见 `+` 几乎不见 `-`。

## L6 · Storyboard 是 XML,能不碰就不碰

**现象**:工具栏按钮实体定义在 `Main.storyboard` 里,直觉上加按钮要改它。
**应用**:上游自己就有"纯代码创建按钮"的先例(ArticleExtractorButton),照抄即可。设置页新行同理 —— 上游在代码里动态增删行(`accounts.count + 1`),跟着这个模式走,932 行的 Settings.storyboard 一个字符都没碰。

## L7 · LLM 的输出永远不能直接信

**现象**(全部真实发生过):把译文包在 ``` 里;把原文和译文一起吐出来(中英对照);整段原样返回不翻译。
**教训**:**提示词是软约束,代码才是硬的。** 提示词里明令禁止的事,模型照样会干。
**应用**:三层防线:①提示词明令禁止;②代码清洗(剥代码围栏、切掉回显的原文);③事后自检(JS 本地判断哪些组还是英文/混了原文,重翻)。每层都要保守设计,宁可漏抓不可误杀。

## L8 · "第一块翻完就点亮按钮"是错误的产品判断

**现象**:我以为提前点亮按钮能传达"已经开始了",实际用户的感受是"按钮说完成了内容却还在变,我不知道该不该等"。
**教训**:**状态指示器的语义必须诚实**:转圈=还在干活,实心=真的干完了。"内容在逐段变中文"本身已经是进行中的信号,不需要按钮抢戏。
**应用**:任何异步流程,完成态只在**全部**结束后设置。

## L9 · 均分的并行块对"顺序阅读"是错的

**现象**:其余内容均分 4 组,每组几千字符 → 用户读完先导块,要干等第 1 组(和最后一组一样大、一样慢),体验断粮。
**教训**:优化目标错了 —— 均分优化的是**总完成时间**,但用户按顺序读,真正该优化的是**第 N 块在被读到之前就绪**。
**应用**:渐进式分组:第 1 组最小(1000 字符),逐组翻倍,4000 封顶。供给节奏跟着阅读节奏走。

## L10 · 标题在正文容器外面

**现象**:只切 `#bodyContainer` 里的内容,标题(`.articleTitle`)漏翻了。
**应用**:改任何"处理正文"的逻辑时,记得问一句:**标题、日期、作者这些容器外的东西要不要一起处理?** 目前只处理标题,日期作者有意不翻。

## L11 · 游离的 Task 不会跟着父任务取消

**现象**:标题翻译用了独立的 `Task {}`,父任务取消时它还活着 —— 用户翻页后,它可能把上一篇的标题译文安到下一篇文章头上(WebViewController 是复用的)。
**应用**:凡是"发射后不管"的 Task:①保存句柄;②`defer { task?.cancel() }`;③任务内部在写回 UI 前检查 `Task.isCancelled`。三样都要。

## L12 · 主题会改 DOM 结构,选择器必须带兜底链

**现象**:8 个内置主题里,Biblioteca 的正文容器是 `body-container` 而不是 `bodyContainer`。只认一个 id 的话,用户换主题后翻译按钮静默失灵。
**应用**:`translation.js` 里所有选择器都是候选链(id → 变体 id → class → 变体 class → 标签名),全落空才报"主题不受支持"。**永远不要静默失败。**

## L13 · Shared/ 的代码会被 macOS 一起编译

**现象**:`TranslationController` 用了 UIKit,iOS 编译过了,macOS 版会挂 —— 而且平时只跑 iOS 根本发现不了。
**应用**:①UIKit 相关文件整个包 `#if os(iOS)`;②**每次改 Shared/ 后必须跑一次 macOS 免签名编译**(命令见 NOTES-progress 第六节)。macOS 常规编译会卡在签名,要加 `CODE_SIGNING_ALLOWED=NO` 系列参数。

## L14 · UIPageViewController 的翻页回调只覆盖手势

**现象**:按钮状态重置挂在 `didFinishAnimating` 上,结果只有手指滑动翻页会触发;右上角箭头、底部"下一篇未读"、列表点选都不走这个回调。
**教训**:修 bug 时别只堵报上来的那个口子 —— 要找**所有路径的必经之地**。
**应用**:挂在 `article` 属性的 `didSet` 上(一切换文章方式最终都经过它)。第二层保险:按钮点击时不信 Swift 记的状态,先问网页当前真实显示的是什么。

## L16 · "替换失败"被静默吞掉,表象是"某组总是最后才翻好"

**现象**:用户报告分块并行时第 1 组总是压轴才变中文,后面的组都好了它还是英文。
排查发现两个叠加机制:
① 组译文替换回页面(applyGroup)失败时,代码写的是 `_ = try? ...` —— 结果被扔掉,
不重试、不计数。这一组一直显示原文,**拖到最后的自检阶段才被救回来**,所以永远最后。
② 引用型博客(Michael Tsai / Daring Fireball,恰是用户的主要订阅)常有一整段
几千字符的 `<blockquote>`,顶层切分切不动它,谁分到它谁就巨慢。
**教训**:异步流水线里**每一步的失败都必须有去向**(重试/计数/上报)。
`_ = try?` 等于把故障扫进地毯下,最后以离奇的表象冒出来。
**应用**:①替换结果并入重试循环;②超大元素下钻一层按子元素切
(同一组绝不跨父节点,否则替换会插错位置);③模型返回裸文本时用原标签包回;
④加了每组"大小/耗时/结果"日志 —— **下次再有性能问题,先看 `[翻译]` 日志,
用数据定位,不猜**。

## L17 · 改提示词必须让旧缓存作废

**现象**:加了"专有名词保留英文"的新规则,但已缓存文章命中缓存直接秒开 ——
旧译文是按旧规则翻的,新规则永远轮不到生效,用户会以为改了没用。
**应用**:缓存键里加了 `promptGeneration` 版本号(TranslationCache.swift),
**每次大改提示词都要把它 +1**,全部旧缓存自动作废重翻。

## L18 · 缓存指纹不能取自"活的"DOM HTML

**现象**:用户报告缓存"时灵时不灵" —— 同一篇翻过的文章,有时秒开,有时明显重新请求。
**原因**:判断"原文变没变"的指纹取的是正文 innerHTML 的哈希。但页面加载后,
NetNewsWire 自带脚本会**异步**改 HTML(图片查看器装饰、懒加载等),
图片加载到哪张取决于用户点按钮的时机 → 同一篇文章两次取到的 HTML 不同 →
哈希不同 → 被误判为"文章更新了" → 重新翻译。
**教训**:**指纹必须取自稳定的东西。** "时灵时不灵"类 bug 十有八九是
输入里混进了时序相关的变量。
**应用**:指纹改为正文**纯文字**(normalizeSpace(textContent))的哈希 ——
文字才是我们翻译的对象,也不受图片装饰影响。同时缓存版本号 +1 清掉坏指纹的旧条目。

## L19 · customView 的工具栏按钮必须有尺寸约束,否则会被压成 0 宽永久消失

**现象**(2026-07-21 用户报告):文章页底部工具栏的"阅读视图"和"翻译"两个按钮同时消失;
一旦出现,之后**所有**文章都没有这两个按钮,重启 app 才恢复。
用户复现方式:反复进文章 → 退回列表 → 再进文章。

**定位过程(值得复用的方法)**:
1. 抓模拟器日志:`xcrun simctl spawn booted log show --predicate 'process == "NetNewsWire"'`
2. 关键证据:`Unable to simultaneously satisfy constraints` 里出现
   `'UIView-Encapsulated-Layout-Width' ButtonWrapper.width == 0` —— 宽度被算成 0
3. 对照消失的是哪些按钮:○☆⌄⬆️(标准 bar button)全在,
   📄阅读视图 和 💬翻译(**两个 customView 按钮**)全没 —— 一个不多一个不少

**原因**:两个按钮都只设了 `frame = 44×44`、没有任何尺寸约束,
宽度靠"图标撑出来的固有尺寸"。而两者在转圈状态都会 `setImage(nil)` ——
图标一没,固有尺寸变 0,iOS 26 的 Liquid Glass 工具栏把它算成 0 宽并**永久塌陷**
(ArticleViewController 是常驻实例,工具栏不重建,所以之后所有文章都受影响)。

⚠️ 阅读视图按钮是**上游自己的代码**,同样的毛病;我照抄它的写法等于复制了隐患。

**修复**:给两个 customView 都加死 44×44 约束
(`translatesAutoresizingMaskIntoConstraints = false` + width/height anchor)。
上游那个按钮的约束加在本 fork 的 `installTranslationButton()` 里,不动上游原有代码行。

**验证(受控对比)**:修复前日志里 `ButtonWrapper ... width == 0` 出现 **40 次**
(其中最早两次在启动过程中、用户尚未交互时);修复后同样的启动路径 **0 次**。

**教训**:
- **把 UIButton 当 `UIBarButtonItem(customView:)` 用时,一定要给显式尺寸约束**,
  永远不要依赖"图标撑出来的固有尺寸" —— 任何会把图标置空的状态都会让它塌掉
- "一旦发生就再也不恢复"这类 bug,几乎都指向**常驻对象上的一次性布局塌陷**,
  找那个不会被重建的东西
- 症状里"哪些坏了、哪些没坏"的**边界**是最强的线索:
  这次"只有两个 customView 按钮消失"直接锁定了原因

## L20 · 本地化:翻译只能写进上游的 .xcstrings,但可以做到"只增行"

**背景**(2026-07-21):把界面完全汉化,需要处理 436 条字符串。

**试过但失败的路**:"完全不碰上游文件,另起一份自己的中文表"。
Xcode 直接拒绝:`Localizable.xcstrings cannot co-exist with other .strings tables with the same name`。
**字符串目录和同名 .strings 不能并存,翻译只能写进上游那个目录。**

**关键手法 —— 复刻 Xcode 的 JSON 格式**:
Xcode 写 .xcstrings 用的格式很特殊(键与冒号间有空格、空对象带空行、**文件末尾没有换行**)。
用普通 `json.dump` 会把整个文件重排,`git diff` 变成"整份文件都改了",
`git pull upstream` 时冲突根本没法读。
`i18n/inject.py` 复刻了该格式,并在写入前自检"能否字节级还原原文件",不能就拒写。
结果:注入 436 条后 diff 只有新增行,逐 key 校验确认原有内容一字节未变。

**教训**:
- **要改一个由工具生成的文件时,先做"读出来再写回去,是否字节一致"的往返测试。**
  一致才动手。这个测试 10 分钟,能省掉后面无数次无法阅读的 merge 冲突。
- 末尾换行这种细节最容易被忽略 —— 往返测试会直接抓出来。
- 注入脚本要**双向报告**:哪些 key 没翻、翻译表里哪些 key 在目录里找不到。
  后者能立刻抓出弯引号/不间断空格抄错,不会静默失败。(实测 436 条零抄错)

**衍生发现**:
- Storyboard 分两类:在 `Base.lproj/` 里的可以纯新增 `<语言>.lproj/xxx.strings` 翻译(零上游改动);
  不在任何 `.lproj` 里的(本项目的 Settings/Account/Add/Inspector)必须先移动文件才能翻。
  判断方法:编译后看它落在 app 包根目录还是 `Base.lproj/` 里。
- 语言选择器的可选项要从 `Bundle.main.localizations` 动态读,**不要写死语言列表** ——
  以后加日语时选择器自动多一行,一行代码都不用改。
- 编译时 Xcode 会自动往 `project.pbxproj` 的 `knownRegions` 加一行新语言,这是必要的,不要还原。

完整操作手册见 `NOTES-i18n.md`。

## L21 · .xcstrings 的"设备变体":译文结构必须和英文一致

**现象**:界面上本该显示"On My iPhone"的地方,汉化后变成了字面的 `我的%@` —— 占位符没被替换。
**原因**:该字符串英文侧用的是 **`variations.device`** 结构
(iphone → "On My iPhone"、ipad → "On My iPad"、mac → "On My Mac"…),
**根本没有 %@ 这个占位符**。我按普通字符串翻成 `"我的 %@"`,自然没有东西可替换。
**教训**:翻译前**先看英文侧那一条是什么结构**。.xcstrings 里除了普通 `stringUnit`,
还有 `variations`(按设备 device / 按复数 plural)。译文必须用同样的结构,
否则不是显示错乱就是原样吐出占位符。
**应用**:`i18n/inject.py` 现在支持在翻译表里把值写成字典,
自动生成 `variations.device` 结构。检查是否还有其它变体条目:
`grep -l '"variations"' **/*.xcstrings`

## L22 · 改上游的界面前,先把「可调的数值」搬进自己的文件

**背景**(2026-07-21):用户要求改文章列表的样子。考古发现列表**不是 storyboard、
不是 Auto Layout**,而是逐帧手算坐标的 manual layout —— 字号、间距、颜色全是
`MainTimelineCellLayout.swift` / `MainTimelineCell.swift` 里的常量。这两个都是上游文件。

**现象(如果不管)**:每调一次字号就在上游文件里改一个数字。改十次就在上游文件里
留下十处散落的改动,`git pull upstream` 时每一处都是潜在冲突,而且冲突长得像
「上游把 12 改成 14,你把 12 改成 16」这种最难判断的样子。

**教训**:**改上游界面代码时,真正该做的第一件事不是改样子,是把「会反复变的数值」

搬运时抄错一个数字,肉眼根本看不出来。做法是找一个**内容固定**的界面做对照 ——

## L23 · 正文页的样式可以「追加一层」,不必改上游的样式表

**背景**(2026-07-21):想改正文阅读页的排版。最直觉的做法是改
`Shared/Article Rendering/stylesheet.css` —— 但那是上游高频改动的文件。

**发现的通道**:`WebViewConfiguration.swift` 里有一个脚本清单
`let filenames = ["main", "main_ios", "newsfoot"]`,这些 .js 会在**每次页面加载前**
自动注入。往这个数组里加一个我们自己的脚本名(**上游改动 = 一行里加一个单词**),
脚本在页面加载完成后往 `<head>` 追加一个 `<style>`,就得到了一层我们完全控制的样式。
上游的 CSS 一个字都不用动,而且对 8 套主题同时生效。

**三个必须知道的细节(都是实测出来的,不是推的)**:

1. **样式必须插进 `<head>`,不能插进 `<body>`。**
   上游 `main.js` 的 `stripStyles()` 会把 `<body>` 里所有 `<style>` 全部删掉。
2. **同优先级时后加的赢,不需要 `!important`。**
   实测:我们的 `.articleTitle a:link { color: #cc0000 }` 盖过了上游同名同优先级的规则。
   这是整条通道的地基,所以专门用探针验证过,没有凭"CSS 应该是这样"下结论。
3. **标题颜色写在 `.articleTitle a:link` 上,不在 `.articleTitle h1` 上。**
   第一次探针写的是 `h1`,颜色没变 —— 因为标题文字被包在 `<a>` 里,
   子元素上的规则赢了父元素。**"没生效"不一定是通道坏了,也可能是选择器选错了对象。**

**方法本身值得复用(就是 L3 的探针法)**:先塞一条**故意难看**的样式
(正文底色刷成琥珀黄),截图确认真的变了,再清空。
如果一开始就写空 CSS,那么"界面没变化"既可能说明链路是通的,
也可能说明链路根本没接上 —— **两种情况长得一模一样,等于没验证。**

**衍生发现**:iOS 版**没有**正文字号滑块(那是 macOS 专属,
`ArticleRenderer.swift` 里 `text_size_class` 包在 `#if os(macOS)` 里)。
iOS 的正文字号来自系统的动态字体设置,由 `styleSubstitutions()` 注入成 CSS 的 `font-size`。
用户想调正文字号,要么改系统设置,要么由我们在这层 CSS 里覆盖。

## L24 · 换 app 图标:别覆盖上游的 PNG,改一行 xcconfig 指到自己的图标集

**背景**(2026-07-21):用户提供了一张新的 app 图标。

## L25 · iOS 26 的图标有三种外观,「色调」那种是系统按亮度生成的

**现象**:换完图标,浅色、深色主屏都好看;把主屏切到**「色调」**外观时,

在 Contents.json 里显式登记成 `luminosity: tinted`。
「色调」外观下立刻变成深底浅字,和系统图标完全一致。**印证了下面这条教训。**

另有一个验证时的坑:改完深色素材后用 `xcrun simctl ui booted appearance dark`
切系统深色模式,图标**没变** —— 不是素材没生效,而是
**主屏图标外观是独立设置**,当时被钉在「默认」上。
测深色图标必须走 主屏长按 → 编辑 → 自定 → 深色。

**教训**:
- **单色图标的观感由原图的明暗分布决定,不由配置决定。**
  想在「色调」模式下好看,素材本身必须是**深底浅字**的高对比灰度图。
- 更一般的:凡是"系统会替你自动生成一个变体"的机制(单色图标、深色模式颜色、
  SF Symbol 的各种渲染模式),**生成质量取决于你喂进去的素材,不取决于你的设置**。

- 验收一个图标必须**三种外观都看**:长按主屏 → 编辑 → 自定 → 默认/深色/透明/色调。

## L27 · 对照物只有在"两种假设会给出不同结果"时才算证据

**这条是我犯了错之后才写对的,过程本身就是教训。**

**现象**(2026-07-21):用户问,真机上 app 图标会跟系统深浅色实时变,
模拟器里为什么不变 —— 是模拟器的问题还是代码没写好?

**我当时的推理**:
1. `assetutil` 确认三种外观都编进去了 ✅
2. 手动选「深色」「色调」,我们的素材都能正确显示 ✅
3. **"决定性证据":Apple 自己的「通讯录」「文件」在深色模式下也是浅色的** ——
   别人家也不跟随,所以不是我们的锅
4. 重启 SpringBoard 后仍不变(排除了"没刷新")

→ 结论:**模拟器不支持自动切换**。

**这个结论是错的。** 用户随后指出:主屏「自定」面板里选「深色」之后,
**下面还有一排子选项「始终 / 自动」**,要选「自动」才会跟随系统。
按用户说的一试,立刻正常 —— 我们的素材一点问题都没有。

**错在哪:第 3 条根本不是证据。**
那条我自以为"决定性"的对照,在两种假设下**预测完全相同**:

| | 假设 A:模拟器不支持 | 假设 B:全局开关没开 |
|---|---|---|
| Apple 的图标会不会跟随? | 不会 | **也不会** |

因为那个开关(当时停在「默认」)作用于**所有** app,Apple 自己的也不例外。
一条对两种假设给出同样预测的观察,**信息量是零**,
我却把它当成了压倒性证据,还因为它"感觉很硬"而停止了继续找原因。

**教训**:
- **对照物有没有用,取决于"两种假设会不会给出不同结果"。**
  动手前先问一句:如果是 A,我会看到什么?如果是 B 呢?
  **两个答案一样,这个实验就别做了,做了也白做。**
- 特别当心**全局性的设置/开关** —— 它天然会让"我的 app"和"别人的 app"
  表现一致,从而伪装成"环境的锅"。凡是"所有 app 都这样"的现象,
  在断定是平台问题之前,先找一遍有没有一个管着所有 app 的开关。
- **"我已经有决定性证据了"这个念头本身就是危险信号。**
  这次我正是因为觉得第 3 条够硬,就没再去翻那个面板的下一层。
- 仍然成立的一条:排查时要分清**"没生效"还是"生效了没刷新"**,
  两者表象一样、修法不同(这次用重启 SpringBoard 排除了后者,这步是对的)。

## L31 · 筛选规则的每个分支都要定义结果,「不处理」不是默认安全的

## L33 · 对限流严格的服务,别为了「提前给点反馈」多打一枪

**现象**(2026-07-21,订阅发现 Phase A):Reddit 搜索能搜出来,**一点订阅就报
「The feed couldn't be found and can't be added.」**。版块名完全正确。

**定位**(值得复用):抓模拟器日志,一句话定案 ——
```bash
xcrun simctl spawn booted log show --last 20m --predicate 'process == "NetNewsWire"' --style compact | grep -iE "FeedFinder|429"
# → response_status=429
```
不是"找不到",是**被限流了**。上游 `FeedFinder` 只判断
`if !response.statusIsOK { throw feedNotFound }`,把 429 翻译成了「feed 不存在」。

**我的两个错**:
1. 搜索时我自己发了一次请求去「验证版块存在」,于是用户一次操作打 Reddit **两枪** ——
   而第二枪(真正要紧的订阅)挨了 429。
2. 那次验证走 `URLSession.shared`,**绕过了 app 自己的 429 记账**
   (`DownloadSession.requestShouldBeDroppedDueToActive429`,按域名退避)——
   只消耗配额、不参与退避,纯负收益。
3. 附带:我自己前面用 curl 连测十几次,模拟器和 Mac 共用出口 IP,
   **额度是我提前打光的**。验证外部 API 时要意识到这一点。

**教训**:
- **省下的请求要留给必须成功的那一步。** 「提前验证」听起来贴心,
  但它抢的是同一份配额;拼错名字反正订阅时也会暴露,而限流造成的**假错误**
  比拼错难查一百倍。
- **自己发的请求要和 app 走同一套下载器/记账**,否则会绕过退避、重试、
  条件 GET 这些既有机制,制造出"单独测都好、合起来就坏"的问题。
- 上游把多种失败压成一个错误时,**在自己这层把话说清楚**。拿不到具体状态码
  就**把两种可能都列出来**并各给一条行动建议 —— 比二选一猜错有用。

## L34 · 图文并排先看是不是表格,别急着怪样式

**现象**:Reddit 帖子正文里,缩略图和文字总是并排,「图片独占一行」的规则不生效。

**真相**(按 L30 的路子,直接拉 `contentHTML` 一眼就看见):
Reddit 把正文塞进了一个**一行两格的表格** ——
`<table><tr><td>缩略图</td><td>正文</td></tr></table>`。
图和文字并排**不是行内流的问题,它们本来就是两个并排的单元格**,
所以任何作用在 `img` 上的 `display: block` 都不可能有效。

**修法**:让这种排版表格的单元格堆叠。选择器**认结构不认来源**:
```css
.articleBody table:has(tr:only-child):has(td > a > img) :is(td, th) { display: block; ... }
```
「只有一行」+「格子里装着被链接包着的图」= 排版用的假表格。

**动手前先量杀伤范围**(这一步不能省):脚本扫全库 196 个表格 →
命中 26 个假表格、170 个真表格零命中,才敢发。
误伤真数据表这种事,**坏了不容易一眼看出来**,必须主动去量。

**顺带堵掉的一个坑**:表格外面套着 `overflow-x: auto` 容器,
所以额外加了一条「表格里的图片不参与顶到屏幕边缘」——
否则负边距会拖出一条横向滚动条。

## L35 · 数据被上游「解析了但没入库」时,绕路重取,别去改数据层

**背景**(2026-07-21,播客语音条):要拿每集的音频地址。

`RSSParser.swift` 里有 `article.attachments.insert(attachment)`。

**真相**:`Article` 数据模型**没有 attachments 字段**,
`ArticlesDatabase` 建库脚本里还有一句 `DROP TABLE if EXISTS attachments`。
**音频地址在入库之前就被丢掉了**,数据库里翻不到。

- ❌ 在数据层加回字段 = 改 `Modules/Articles` + `Modules/ArticlesDatabase`(**C 级禁区**),

- ✅ **要用的时候按 `Article.feedID` 重拉一次 feed**,用**上游自己的解析器**

- 按 feed 缓存,一个 feed 一次会话只拉一次
- **连「这个源根本不是播客」也要缓存**(负缓存)。没有负缓存的话,

**教训**:发现「上游解析了但没存」时,先算一笔账 ——
**重新取一次的成本**(加缓存后往往很低)通常远小于**改数据层的长期代价**。

## L36 · 同一个 API 的两个字段,可能一个能用一个是残次品

**现象**:iTunes 的单集接口同时返回 `episodeGuid`、`trackId`、`episodeUrl`。

- 混合方案往往比纯方案好:**权威数据取自权威来源,附加信息取自方便来源**。

## L37 · 网页的"身份"被上游顺手设成了第三方域名 —— 嵌入的第三方内容会拒绝工作

**现象**(2026-07-21):在正文里嵌 YouTube 播放器,**所有视频**都报
「此视频不能观看 错误代码 152」。

**排查过程(这条的价值主要在过程)**:

先用**服务端可验证的手段**逐个排除,每一条都有证据、都不靠推测:

| 假设 | 怎么验的 | 结果 |
|---|---|---|
| 作者禁止嵌入 | oEmbed 接口测用户订阅里 8 个真实视频 | ❌ 8/8 都允许嵌入 |
| UA 被 YouTube 拒 | 分别用 Safari UA 和 app 的 NetNewsWire UA 请求 embed 页 | ❌ 两者都正常返回播放器 |
| 视频 CDN 被拦截规则挡了 | 把 ContentRules.json 全部 42 条打印出来看 | ❌ 没有 googlevideo/ytimg/youtube |
| Referer 不对 | 三种 Referer 各请求一次 | ❌ 都 200 |

到这里服务端手段用尽,剩下两个假设都在客户端。于是设计**一个对照实验**:

> 把**同一个视频**插进一篇非 YouTube 的文章里(The Conversation)。
> 同样的代码、同样的视频,**唯一的变量是文章的身份**。

结果:那边能正常播放 → **一次就定案**。

**真正的原因**:上游用 `loadHTMLString(html, baseURL:)` 渲染文章,
baseURL 取自文章链接。而 YouTube 文章的链接正好是
`https://www.youtube.com/watch?v=…` —— 于是**文章页自称是 youtube.com**。
嵌进去的播放器一校验"谁在嵌我",看到一个自称 YouTube 却没有 YouTube 会话的
页面,就拒绝播放。

**修法**:只对 YouTube 文章把 baseURL 换成中性身份。安全性是**量过的** ——
全库 15 篇 YouTube 文章里正文非空的有 **0 篇**,而 baseURL 的正经用途就是
解析正文里的相对链接,所以不可能解析错;另外 2293 篇走原分支不受影响。

**教训**:
- **`loadHTMLString(_:baseURL:)` 的 baseURL 不只是"解析相对链接的前缀",
  它决定了这个网页对外的身份(origin / Referer)。** 当 baseURL 恰好是某个
  第三方站点时,从该站点嵌入的内容会看到一个"冒名"的宿主页面而拒绝工作。
- **排查顺序:先穷尽服务端可验证的假设**(curl 能测的:状态码、UA、Referer、
  是否被拦),**再设计客户端对照实验**。前者便宜且证据硬,后者贵但能定案。
- 对照实验必须满足 L27 的标准:**两个假设要给出不同预测**。
  这次"同一个视频换一篇文章"正好只改变一个变量,所以一次就够。
- 改动这类全局代码路径时,**打击面要用数据量出来**,不能只说"逻辑上只影响 X"。

Ars Technica 的鲜橙色则被算成灰褐色。

**真相**:代码里用的是 `IconImageCache.shared.imageForFeed(feed)`。它的取图顺序是
「feed 自带图标 → favicon → **SmallIconProvider**」,而最后那一档
(`Feed.smallIcon` → `FaviconGenerator.shared.favicon(feed)`)会
**合成一张地球仪占位图,颜色来自 `ColorHash(feed.url)` —— 网址的哈希值**。
真图标还没下载好的那一瞬间调用它,拿到的就是这张紫色地球仪。
而列表行是稍后渲染的,那时真图标已到,所以行里是墨绿的 —— 一个界面里两种颜色,
看起来像"颜色算错了",其实是"两次取图取到了不同的东西"。

**第二层问题(我自己写的)**:拿到占位图后我照常渲染并写了 `renderedKey`,
于是"图标就绪"通知来时,补装逻辑因为 `renderedKey != nil` 直接跳过 —— **永远卡在占位色**。

**修法**:
- 取图改成直接问真图标的下载器:`FeedIconDownloader.shared.icon(for:)` →
  `FaviconDownloader.shared.favicon(for:)`。它们没缓存时会**顺带发起下载**,
  到货后通知补装。**绕开 imageForFeed,就绕开了占位图那一档。**
- 拿不到真图标时**故意不设 `renderedKey`**,让通知能把它救回来。

**教训**:
- **"取图/取值"的 API 常常带一条你没注意的兜底分支**,而兜底值往往是"看起来正常、
  实际无意义"的东西(合成占位图、默认色、空对象)。用之前先读完它的取值顺序,
  尤其要问:**最后一档返回的是什么?**
- **同一份数据在界面上出现两次却长得不一样(这里是行图标 vs 头图),
  几乎必然是两条取值路径**,而不是"渲染错了"。先去比对两边调的是不是同一个 API。
- 「先用兜底值渲染,等真值到了再刷新」这个模式里,**必须让"用的是兜底值"这件事
  可被后续逻辑看见**(这里是不写 renderedKey),否则兜底值会永久固化。

## L52 · 我连续两次认定了错误的根因,而"把图导出来看一眼"一次就定案

**这条记的是排查方法本身,过程比结论值钱。**

上面 L51 那个"墨绿被显示成紫色"的问题,我在找到真凶之前,**先后给出过两个错误诊断**:

| # | 我的判断 | 依据 | 为什么错 |
|---|---|---|---|
| 1 | 半透明像素的**预乘 alpha** 把主色投票带偏了 | 机制上完全成立,而且确实会让颜色变暗变灰 | 用真实图标做离线对照后发现:两种 alpha 门槛算出的主色**完全一样**。机制成立 ≠ 它就是本次的原因 |
| 2 | `CGContext(data: &buffer, …)` 是**未定义行为**,读到了垃圾内存 | 这是真 bug(Swift 数组指针逃逸),而且"结果随机"完美解释"颜色莫名其妙" | 改成 `withUnsafeMutableBytes` 后重编重装,**主色一个字节都没变**,还是那个紫色 |

第三步我才做了本该第一步做的事:**把 app 实际拿到的那张素材图 `pngData()` 写进 Caches,
用命令行拷出来打开看** —— 一眼看见是一张**紫色地球仪**,当场定案。

**教训**:
- **L30 说过"这东西是谁画的,是可以直接查的事实,不是需要推理的问题"。这次我又忘了。**
  凡是"某个值不对"的排查,**先想办法把那个值本身导出来看**(存文件、打日志、写进数据库),
  再去推理它为什么不对。前者一步,后者可以无限步。
- **"这个机制确实会造成这种现象"不等于"这次就是它造成的"。** 两次误判都栽在这里:
  预乘 alpha 会让颜色变灰(真的)、野指针会让结果随机(真的),但它们都**没有发生**。
  这是 L27 的同一条规则换了个马甲 —— 判断一个假设有没有价值,要看**它能不能被证伪**,
  而不是看它听起来多合理。我当时该做的是:改完之后先验证颜色变了没有,再宣布修好。
- 具体到图像类问题:**把中间产物落盘**(素材图、遮罩、合成结果)是最快的排查手段,
  几行代码,比任何推理都硬。用完记得删掉那段临时代码。

## L53 · 异步到货的数据,不能用「问一次就算数」的方式去取

**现象**(2026-07-22,头图抓高清图):硅谷101、Marques Brownlee、Links TV 三个源
始终退回纯色渐变。而实测它们的大图明明存在:og:image 分别是
**1400×1400 的播客封面**和 **900×900 的 YouTube 频道头像**,
连元数据都**已经缓存在本地数据库里**。

**三层原因,全都源自同一个误解:我把「异步到货的数据」当成「同步就位的数据」在用。**

1. **候选来源里少了 og:image。** 这三个源的 `iconURL` / `faviconURL` **全是空的**
   (查数据库确认),只能靠网页元数据里的 og:image。补上后**零额外请求** ——
   元数据本来就是上游缓存好的。
2. **`HTMLMetadataDownloader.cachedMetadata(for:)` 是个内存缓存,app 启动时是空的。**
   它的语义是「有就同步给你;没有就**异步去取并返回 nil**,取到后发
   `.htmlMetadataAvailable` 通知」。所以**第一次抓图时,apple-touch-icon 和 og:image
   这两类候选根本还不存在**。我却按「问过了、没有」处理,记进负缓存,永不重试。
3. **修好重试之后仍然不灵 —— 一个竞态**:元数据到货的通知,**正好落在第一轮抓取
   还在进行的那零点几秒里**,被 `inFlight` 闸门挡掉;而这轮结束后不会再有通知,
   于是永远轮不到第二轮。
4. **还有一个「知足」陷阱**:硅谷101 第一轮能拿到一张 192px 的 apple-touch-icon,
   达到了最低门槛,于是被当成最终答案缓存 —— 那张 1400px 的封面就永远没机会了。

**修法(四条对应)**:①候选链补 og:image(放最后,够大的图标会早停,不打扰正常的源);
②负缓存改成**尝试计数**,允许新信息到货后再试;③抓取途中收到通知就**记一笔,
本轮结束后自动再抓**;④判据从「够门槛就收工」改成「**没到理想尺寸就继续升级**」,
新的一轮拿到更大的才替换。四条缺一不可 —— 我是一条条补上去的,每补一条都以为好了。

**教训**:
- **看到 `cachedXxx()` 这种同步取值 API,先去读它「没有的时候干什么」。**
  如果答案是「异步去取 + 发通知」,那么**第一次调用注定拿不到**,
  你的代码必须为「第二次」留好路 —— 而这条路上通常还埋着 in-flight 竞态。
- **「够用就收工」在数据会变好的场景里是错的。** 当更好的答案可能稍后才到,
  判据要写成「**离理想还差多少**」,而不是「有没有过线」。
- **一个功能失灵可能是好几个独立原因叠加的**。这次连修四处才通。
  每修一处都要**实际验证一次**(我靠日志逐轮确认候选数从 3 → 4、尺寸从 0 → 900px),
  否则很容易在只修了一半时误以为完工。

## L54 · 调"颜色的深浅"要在 HSB 里走明度轴,别在 RGB 里朝黑白混合 —— 那会把饱和度一起冲掉

**现象**(2026-07-22,头图标题染主题色):按主题色给标题上色,深色模式下用户说
「看着不太协调」。

**用户说得很准,而且原因是可量化的**:

| | 色相 | 饱和度 | 明度 |
|---|---|---|---|
| Acquired 的主题色 `#052C29` | 175° | **0.89** | 0.17 |
| 实际渲染出的标题色 `#B6C2C1` | 175° | **0.06** | 0.76 |

色相一点没变,**饱和度从 0.89 掉到 0.06** —— 浓墨绿被洗成了一块中性灰。
于是头图是浓浓的墨绿、标题却是灰的,两者看不出关系,这就是"不协调"的来源。

**根因**:第一版的写法是「在 RGB 空间把品牌色朝白色(深色模式)或黑色(浅色模式)混合,
直到对比度达标」。**朝白混合 = 三个通道一起往 255 靠 = 通道差值被压缩 = 饱和度必然下降**。
朝黑混合同理。也就是说,这个方法在"提高对比度"的同时**必然摧毁颜色身份**,
两个目标天然打架 —— 而我当时只盯着对比度这一个指标,还专门写了脚本证明"全部达标"。
**指标全绿,东西却难看**。

**修法**:改到 HSB 空间,**锁住色相与饱和度,只沿明度轴走**,一步一步走到对比度达标为止。
同一批源改完后:Acquired 深色模式下是 `#45918B`(饱和度 0.53),是真的墨绿了。

**教训**:
- **"变深 / 变浅"这个操作,语义上属于明度轴,就应该在有明度轴的色彩空间里做**(HSB/HSL/LCH)。
  在 RGB 里朝黑白混合是个看起来等价、实则会顺手改掉饱和度的近似做法。
- **一个只用单一指标(对比度)验证过的改动,可能在别的维度上已经坏掉了。**
  我上一轮跑了 7 个颜色 × 2 种模式的对比度表,全部达标,于是宣布完成 ——
  但没有任何一项检查在看"它还是不是原来那个颜色"。
  **补救办法很简单:把改动前后的 HSB 三个分量一起打出来对比**,一眼就能看见饱和度崩了。
- 用户说「不协调 / 怪」这类模糊反馈时,**先把可量化的属性列出来逐个对比**
  (色相、饱和度、明度、对比度),往往一眼定位,比反复猜"哪里怪"快得多。

## L55 · 「升级」逻辑只比了一个维度,于是用更大的坏东西换掉了更小的好东西

**现象**(2026-07-22 用户报告):进 Six Colors 时**头图一闪而过**,然后就没了。

**日志一路看下来,过程很清楚**:

```
① 144px 兜底 → 主色渐变
② 抓到 256px 的 sixcolors.png → 用大图,非白像素 39% ✓   ← 用户看到的"一闪"
③ 元数据到货 → 再抓一次
④ 抓到 1910px 的 og:image → **因为更大,替换掉了 256px**
⑤ 这张 og 图是白底的,非白像素仅 18% → 低于门槛 → 退回主色渐变
⑥ 而它的主色是 #E6E6E6(近白)→ 和纸色 #F3F0EB 几乎一样 → 头图等于消失
```

**根因**:我在 L53 里加的「不够好就继续升级」判据,**只比像素数**。
于是「更大」压过了「能用」——一张又大又白的 og 横幅顶掉了一张小而合格的 logo。
**而"能不能用"这个判据我明明已经写好了**(非白像素占比),只是没有用在选图那一步,
只用在了最后决定"这张图铺不铺得开"上。**同一个判据,两处口径不一致。**

**修法两条**:
1. 选图的分数改成 `像素 × 方图系数 × 可用系数`,可用系数取 **0.05**(近乎分类优先)。
   实测:256px 的 logo 得分 **256**,1910px 的白底 og 图得分 **29**,不会再被顶掉。
   替换判定也从「像素更多」改成「分数更高」。
2. 退回主色渐变时,保证那个颜色**在纸色上看得见**(`visibleGradientColor`,目标对比度 1.5)。
   否则近白的主色会让整片头图直接消失 —— 这是独立于上面那条的第二重保险。

**教训**:
- **凡是「用新的替换旧的」的逻辑,判据必须和「接受这个东西」的判据是同一套。**
  这次是同一个属性(非白像素占比)在两个地方各用各的,于是"选进来"和"用得上"打架。
  写升级/择优逻辑时先问一句:**我拿来排序的这个分数,包含了下游会拒绝它的那些理由吗?**
- **单维度择优必然踩这个坑**。"更大 / 更新 / 更快"都只是代理指标,
  真正要的是"更合用"。加权时宁可把否决项压到近乎分类优先,也别让它和主指标线性权衡。
- 这类 bug 的表现是**闪一下就没**(先对后错),很有辨识度:
  **说明正确结果曾经出现过,是被后续步骤覆盖掉的** —— 直接去查"谁在它之后写了这个值"。

## L57 · iOS 26 的 `navigationBar.bounds` 把状态栏也算了进去 —— 拿它的中点对齐会偏高一大截

**现象**(2026-07-22,做标题飞向导航栏的动画):标题一路飞过头,**眼看要飞出画面**。

**定位手段(可复用):直接在用户的截图上量像素。**
- 两个圆钮(返回/筛选)的中心:**85pt**
- 标题被算到的位置:**57pt**
- 差了 28pt,而且 57 ≈ 114/2 —— 于是立刻能反推出:
  我用的 `navigationBar.bounds` 高约 **114pt**,即**从屏幕最顶端算起、把状态栏一起包进去了**,
  它的"正中"自然落在状态栏中间偏下,而不是圆钮那一条。

**修法**:改用两个可靠的量算停靠区 ——
`状态栏高度`(`windowScene.statusBarManager.statusBarFrame.height`)到 `safeAreaInsets.top` 之间那一条。
安全区顶 = 状态栏 + 导航栏,所以这一条正是导航栏本身,中点 = (62+106)/2 = **84pt**,与实测的 85pt 吻合。
另加一道保险:插值结果再怎么算,都不许越过停靠区上沿(`max(interpolatedY, minCenterY)`),
免得几何一变又飞出屏幕。

**教训**:
- **系统控件的 `frame`/`bounds` 未必等于"你眼睛看到的那块"。** 导航栏尤其如此:
  它的边界随系统版本、大标题模式、iOS 26 的浮动按钮样式而变。
  要对齐"看得见的东西",优先用**语义明确的量**(安全区、状态栏高度),
  少用控件自身的几何。
- **在用户给的截图上量像素,是最快的定位手段。** 这次量出 85 与 57,
  再看出 57≈114/2,一步就锁定了原因 —— 比读文档猜导航栏布局快得多。
- **凡是"按插值把东西移到某处"的动画,都要给终点加一道硬性夹取。**
  几何算错时,夹取能把"飞出屏幕"降级成"停在边界上",不至于完全失控。

## L58 · 脚本改代码出错,把递归悄悄种进了函数里 —— app 一启动就栈溢出崩溃

**现象**(2026-07-23 用户报告:app 打不开,附崩溃日志):
`Thread stack size exceeded due to excessive recursion`,栈里
`nnwInstallNavigationBarAppearanceUpdater` ↔ `nnwRefreshNavigationBarAppearance`
互相调用套了 **47431 层**。app 一进文章页(其实是 viewDidLoad)就崩。

**根因**:上一轮我用一个 python 脚本改这两个方法,脚本里有两处 `replace`:
- 第一处想把 `nnwRefreshNavigationBarAppearance` 的实现换成暖纸色 —— 它匹配的是"那五行 appearance 代码",
  但**那五行在 viewDidLoad 和这个方法里都出现过**,`str.replace` 默认替换所有匹配 → 把
  viewDidLoad 里也顺手插了一行"钩子",又把 refresh 末尾加了一行调回 install。
- 第二处 replace 因为锚点对不上直接抛了 AssertionError,**但第一处的破坏已经写进文件了**。
- 我看到 AssertionError 后只补了 install 里缺的那行,**没有回头检查第一处到底改了什么**,
  于是 refresh→install→refresh 的循环留在了文件里。而且编译能过(递归不是编译错),
  装机也不崩(时间线页不触发),直到用户点进文章页才爆。

**修法**:重写这两个方法,把分工写死 —— install 只"装一次+注册",refresh 只"重建外观",
两者绝不互相调用,并在注释里明写这条禁令。

**教训**:
- **`str.replace` 不带 count 参数时替换所有匹配。** 改代码的脚本里,要替换的片段但凡可能
  在文件里出现多次(样板代码、重复的几行),必须先确认唯一性,或改用带唯一上下文的锚点、
  或限制 `count=1`。这次那"五行 appearance"正是典型的会重复的样板。
- **脚本抛异常时,不能假设"什么都没改"。** Python 的 `open(w)` 是即时落盘的,
  异常前已经写入的部分留在了文件里。出异常后第一件事是 `git diff` 看**实际**改了什么,
  而不是只补上"看起来还差的那一步"。
- **编译通过 + 装机不崩 ≠ 没问题。** 无限递归、只在特定页面触发的崩溃,都能躲过这两关。
  改完动了行为的代码,至少要走一遍**真正会执行到那段**的路径(这次是"进文章页")。
  我改的是文章页导航栏,却没进文章页看一眼,是这次的直接疏漏。
- 这也是 L1 那类"改上游/样板代码"风险的又一面:**样板越像,脚本越容易误伤**。

## L59 · registerForTraitChanges 在 UIPageViewController 子树里不触发;与其"变了重建",不如让外观天然自适应

**现象**(2026-07-23,第二次报"深色模式文章页顶栏发浅"):明明上一轮"修好"了,又复发。

**日志一句话定案**(L52 的正确应用 —— 先暴露中间状态,别猜):
给 `nnwRefreshNavigationBarAppearance` 加一行日志打印 style,然后**停在文章页切深色**。
结果:切深色后**没有任何新的 refresh 日志**,只有进文章时那条 `style=light`。
→ 确认 `registerForTraitChanges([UITraitUserInterfaceStyle.self]){…}` 在这个场景**根本没触发**。
(文章页是 `UIPageViewController` 的子 VC,这类嵌套子树的 trait 变化传播不总是到达每个 VC。)

**为什么上一轮以为修好了**:我上一轮的验证方式是"**先切深色、再进文章页**" ——
那条路径 viewDidLoad→install→refresh 在深色下跑,自然是深色,看起来对。
真实会坏的场景"**停在文章页原地切深浅色**"我没测,于是假阳性过关。
(这和 L39 同源:测试路径没覆盖真正会出问题的那条。)

**真正的修法**:不要"变了重建",而是让外观**天然自适应**。
把 `navigationItem` 的三个 appearance 设成 **nil** → 回落到系统内置默认导航栏外观,
那个是永久深浅自适应的毛玻璃(订阅列表页就是它,因为压根没设过 appearance)。
设一次就够,不依赖任何触发。上游 viewDidLoad 里那句 `configureWithDefaultBackground()`
反而是"一次性固化、不自适应"的根因 —— 我们设 nil 把它覆盖掉。

**教训**:
- **`registerForTraitChanges` / `traitCollectionDidChange` 的触发不是无条件可靠的**,
  尤其在容器 VC(UIPageViewController/UISplitViewController)的子树里。
  依赖"某某变化时回调我"来维持外观一致,是脆的。
- **能让系统自适应的,就别自己缓存+重建。** 设 nil/用系统默认,往往比"监听变化手动刷新"
  更省、更稳 —— 少一条会漏触发的链路。手动重建只在系统确实不自适应时才做。
- **"修好了"必须用会真正复现 bug 的那条路径验证**,不能用一条恰好绕开 bug 的路径。
  这次两轮都栽在"先切深色再进页面"这条绕开路径上(L39 的又一次)。

## L60 · 透明导航栏会露出背后 WKWebView 的底色 —— web 深浅色不跟 app 同步时,浅色顶栏会变黑

**现象**(2026-07-23):想让文章内容页顶栏更"透",把 `scrollEdgeAppearance` 设成
`configureWithTransparentBackground()`(完全透明)。深色模式下看着挺透(用户说"非常透"),
**但切到浅色模式,顶栏变成一片纯黑**(用户截图),而正文是浅纸 —— 顶黑身浅,很割裂。

**原因**:导航栏透明后,它那一条区域**露出的是背后的 WKWebView**。而 web 正文的深浅色
走的是网页自己的 `prefers-color-scheme` / 注入的 CSS,**不总和 app 的 userInterfaceStyle 同步**
(尤其 app 切了深浅色但 web 没重新渲染时)。于是浅色 app + 深色残留的 web = 透明顶栏露出黑底。
"完全不透"和"变黑"其实是同一件事:透出来的是黑的。

**教训**:
- **透明导航栏 = 把顶栏那块的观感外包给了背后的视图。** 只有当背后是**你能控制、且一定
  和 app 同步**的内容时,透明才安全(比如订阅列表页背后是 UIKit 列表,深浅色系统统一管)。
  背后是 WKWebView(内容深浅色自成一套)时,透明会露出不受控的底,深浅色一错位就翻车。
- **"某方案在深色下好看"不等于"它对"** —— 这次深色下透出深 web、看着透,恰好掩盖了
  "透出的是 web 底"这个事实;换浅色立刻暴露。视觉方案必须深浅两色都验过才算数(又一次 L59)。
- 回退到「系统默认毛玻璃(全 nil)」是这一串尝试里唯一深浅都不出 bug 的:毛玻璃是**不透明**
  地盖住背后,所以不受 web 底色影响。代价是不如透明"透"——这是稳定性和通透感的取舍。

## L61 · 顶栏「不够透」的真凶:普通标题栏的 scrollEdgeAppearance 设 nil 会回落成毛玻璃

**现象**(2026-07-23):文章内容页顶栏怎么调都"偏实、不够透"。前面几轮换过材质、
改过颜色、试过完全透明,都没做出订阅列表页那种"顶部通透、滚动渐显毛玻璃"的两态。

**根因,两层,缺一层都做不成**:

**① 系统给的默认值本身就没有"透明态"。**
iOS 15 起,**普通标题栏**(不开大标题)的 `scrollEdgeAppearance` 若为 `nil`,
会**直接回落成 `standardAppearance`** —— 于是"内容在顶部"和"内容滚起来"用的是
**同一套毛玻璃**,永远不可能透。
而订阅列表页之所以零代码就有两态,是因为它 `prefersLargeTitles = true` ——
**大标题模式下系统给的 scrollEdge 默认值才是透明的**。
→ 所以「照抄订阅列表页 = 什么都不设」是错的,那是场景不同带来的假象。
文章页必须**显式**把 scrollEdge 设成 `configureWithTransparentBackground()`。

**② 系统不知道该盯哪个滚动视图,于是一直按"已经滚了"处理。**
两态切换的前提是导航栏能读到"内容滚到哪了"。订阅列表页背后就是普通列表,系统自己找得到;
文章页的 WKWebView 藏在 `UIPageViewController` 的子页面里,系统找不到。
→ 用系统正规接口 `setContentScrollView(_:)`(iOS 15+)把线接上。
调用时机要两处:**网页加载完**(本页第一次就绪)+ **翻页动画结束**
(翻到的那页可能早就加载完了,不会再触发前者)。

**教训**:
- **"某某属性设 nil = 用系统默认"这个直觉,在 UINavigationBarAppearance 上是错的。**
  nil 在这里的语义是"回落到另一个 appearance",不是"回落到系统内置外观"。
  同理:大标题模式和普通模式的默认值**不一样**,把一个页面的"零代码就好看"
  搬到另一个页面前,先确认两边的模式是否相同。
- **"抄那个好看的页面"要抄到机制层,不能只抄表面。** 这一串折腾里,
  「订阅列表页没写任何代码」这个观察是对的,但由它推出的「所以我们也不该写代码」是错的 ——
  同样的效果在不同模式下,一个来自默认值,另一个必须手写。
- 系统级两态(透明 ↔ 毛玻璃)一旦接通,**过渡动画、深浅色、材质全是系统的**,
  比自己拿 alpha 去插值一层毛玻璃更稳、更像原生(自绘那条路见 L56 的层级坑)。

## L62 · 毛玻璃要"薄一点",得调浓度(animator 的 fractionComplete),不是调 alpha

**现象**(2026-07-23):文章列表页停靠时那条毛玻璃,用户连着两轮说"不够透"。
中间换过材质(chrome → thin → ultraThin,已经是系统最透的一档),仍然嫌厚。

**先用一个能分辨真假的实验定性,别再猜**(L27 的应用):
当时有两个都说得通的假设 —— ①毛玻璃坏了(降级成半透明白板);②毛玻璃好的,但背后
那几行**已读**文章只有 45% 浓度(`readAlpha`),淡灰字被模糊+提亮冲没了。
两个假设会在同一个场景给出**不同结果**,所以让用户把**未读**文章(100% 浓度的深色粗体标题)
滚到顶栏后面截一张图:字透过来了 → 假设②。一张截图定案,没有多花任何一轮。

**真正的病根:拿 `alpha` 当"厚度"旋钮使。**
苹果明确说 `UIVisualEffectView` 的 alpha 小于 1 会让模糊失真。更要命的是**观感**:
alpha 调出来的"薄"= 清晰内容和模糊内容**叠加**的重影,不是真的薄。
所以以前只能在"alpha=1 的完整毛玻璃(太厚)"和"重影"之间二选一。

**正确做法**:把「从无到毛玻璃」这个变化交给 `UIViewPropertyAnimator`,
再用 `fractionComplete` 把它**停在任意档位**,alpha 全程保持 1。
那是系统插值出来的真毛玻璃 —— 模糊半径和着色一起按比例减弱。
于是"厚度"第一次成了一个可以随便调的数(`TimelineStyle.headerDockedScrimStrength`)。

**⚠️ 这个动画器有个会崩的坑,写的时候必须一起处理**:
`UIViewPropertyAnimator` 停在 `.active`(暂停中)状态时被释放会**直接崩溃**。
而本工程是 Swift 6 + 主线程隔离,**`deinit` 里够不着它**。收尾只能放在
`willMove(toWindow:)`(页面离开窗口时停掉)+ 回到顶部时顺手停掉 —— 让它活着的时间尽量短。
另外:`pausesOnCompletion = true` 不能少,否则进度推到头它就自动作废,之后再喂进度毫无反应。

**教训**:
- **"把某个视觉效果调淡"的第一反应是改 alpha,但对系统材质类视图(毛玻璃、振动效果)
  这条路是错的。** 它们有自己的强度概念,要用系统给的插值机制去调。
- **换材质换不出来的东西,别一直换材质。** 这次在三种 material 之间来回换了几轮都没解决,
  因为可调的那个维度压根不在材质列表里。**发现"选项试遍了都不对"时,
  要怀疑的是维度选错了,而不是继续在同一维度里挑。**

## L63 · 两个各自正确的功能,共用同一个状态就成了死循环 —— 滚动藏栏 × 顶栏跟踪滚动 = 栈溢出

**现象**(2026-07-23 用户实测):文章内容页往下滑,**卡死几秒后闪退**。
崩溃报告里 **28000 层递归**,`EXC_BAD_ACCESS` 撞在栈保护页上。

**回路**(从崩溃栈一层层读出来的,不是猜的):
```
滚动回调 → 藏栏 → 导航栏收起 → 安全区变化 → 系统调整滚动位置
        → 位置变了又发滚动回调 → 判断方向 → 现栏 → 安全区又变 → ……
```
两头都在改同一个东西(安全区/滚动位置),于是来回拉锯,永不收敛。

**为什么以前不崩、这次崩了**:「滚动藏栏」上线时这个回路就已经存在,只是闭合得松。
这次为做顶栏两态,把网页滚动视图交给导航栏跟踪(`setContentScrollView`),
导航栏于是会**主动**去更新那个滚动视图的安全区 ——
崩溃栈里 `_updateViewSafeAreaInsetsAndEagerlyUpdateContentScrollView` 这一帧就是铁证。
回路一下子闭合得又快又紧,几秒就爆。

**修法**:给藏/现栏加一道**防重入闸门**(`isTogglingBars`),切换期间所有滚动回调原地返回;
闸门到**下一轮 runloop** 才抬起(切换引发的布局是本轮同步跑完的,必须整段罩住);
抬闸时顺手 `primed = false` 重新取基准,免得把"系统的位置调整"当成用户滑动。
并把 `hideBars`/`showBars` 收进唯一入口 `nnwToggleBars(hide:)`,杜绝有人绕过闸门。

**离线复现 + 验证(这次真正学到的东西)**:
我把这段判断逻辑连同"切换→安全区变→同步回调"这条回路写成一个几十行的模拟,
扫 667 种滑动组合(安全区变化量 × 每帧滑动距离):
**旧代码 57 种爆栈**(爆栈参数正好是"安全区变 90~95pt、每帧滑 46pt",
即导航栏+状态栏的真实高度 + 快速滑动),**加闸门后 0 种,最深 2 层**。

**教训**:
- **两个功能各自都对,耦合到同一个状态上就可能成环。** 判断标准很简单:
  **A 改的东西会不会触发 B?B 改的东西会不会触发 A?** 只要两个"会",就必须有闸门。
  凡是"在滚动回调里改布局(藏栏/改 inset/改 safeArea)"的代码,**默认就该防重入**。
- **我又一次违反了 L58 的纪律**:编译通过、装机启动、日志无异常就交付了,
  但**滚动这条路径根本没被走到**。改的是"滚动时的行为",却没有任何一步真的滚动过。
  → 以后凡是我(AI)无法亲自操作的交互路径(滚动、拖拽、手势),
  **必须把逻辑抽出来做离线模拟**,而不是把风险丢给用户去撞。
  这次的模拟只花了几分钟,却精确复现了崩溃并证明了修复 —— 早写一轮就能省掉一次闪退。

## L65 · 拖放占位存在时改数据源 = 必崩;以及「订阅源顺序」在上游模型里根本不存在

**现象 A**(2026-07-23 用户实测):把源拖到收起的文件夹上、等它弹簧加载展开时闪退。
崩溃栈直指 `expandFolder` 里的 `dataSource.apply`,断在 UIKit 的
`_endItemAnimationsWithInvalidationContext:tentativelyForReordering:`(批量更新校验)。

**根因**:`.insertAtDestinationIndexPath` 这个落点意图会让 UICollectionView
**在列表里插入一个占位空隙**(就是那条"让开的缝",落点反馈全靠它)。
占位不在 diffable 的快照里,所以此时再 apply 新快照,UIKit 校验行数时对不上 → 断言失败。

**修法**:两种意图分场景用 ——
**悬停在文件夹行上用 `.insertIntoDestinationIndexPath`(不插占位,可安全展开);
其余用 `.insertAt`(要那条让位动画)**。
而弹簧加载**只对文件夹计时**,两处判断又都基于同一个"悬停对象",
于是天然满足不变量:**会触发展开的状态,一定不插占位**。(离线把这个不变量的 5 种状态跑了一遍。)

**教训 A**:
- **拖放进行中,UICollectionView 的实际行数可能比数据源多一个(占位)。**
  凡是"拖放途中改数据源"的功能(弹簧加载展开、自动新建分组…),
  都要先确认当前落点意图**不带占位**,否则必崩。
- 这类崩溃**编译期和静态阅读都看不出来**,而且只在特定手势路径上出现 ——
  又一次印证:交互路径必须让用户实测,或者把不变量抽出来离线验(L63)。

---

**现象 B**(同一轮,用户要求"能不能拖动调整源的先后顺序"):**做不了,而且不该做。**

考古结论(证据确凿,别再重新评估一遍):
- `Folder.topLevelFeeds` 和 `Account.topLevelFeeds` 都是 **`Set<Feed>`** ——
  集合本身**无序**,模型层压根没有"顺序"这个概念可存。
- 上游列表的顺序是**显示时现算的**:`SidebarTreeControllerDelegate` 里
  `sortedAlphabeticallyWithFoldersAtEnd()`,**强制按字母排、文件夹排在后面**。
- 所以即使我们在管理页自己存一份顺序,**主列表(用户真正读文章的地方)照样按字母排** ——
  两个页面顺序对不上,反而更糊涂。OPML 导出/导入、同步账户也都不会带上这个顺序。

**上游的设计意图是「用名字控制顺序」** —— 用户自己的文件夹就是这么起名的
(`01 宏观`、`02 社会`、`03 严肃长文`…),源要排序照做即可。

**教训 B**:
- **接到"能不能拖动排序"这类需求,先查模型里到底存不存顺序。**
  存在 `Set` 里就是明确答案:不存。这时候硬做,等于在 UI 层伪造一个
  底层不承认、别处也不认账的状态。
- **给不出功能时要把替代路径一起给出来**(这里是"用名字前缀排序"),
  否则用户只听到"做不了"。

## L66 · 对有内部状态机的系统 API,**调用顺序本身就是语义** —— 一行位置放错,功能整个失效

**现象**(2026-07-23,拖动排序做完后用户复测):
「同一区域里拖动改变不了顺序了」「拖到文件夹第一行,却出现在第三行」。
用户原话:「我不太能完全总结出具体问题,就是拖动是有问题的,落点不准,或者拖动无效」。

**根因:一行代码放错了位置。**
上一轮为了修"预览缩回原点"的动画,我加了
`coordinator.drop(dragItem, to: previewTarget)`,并且**写在了所有数据计算之前**。
而这个调用会**当场改动 collectionView 的内部状态**;
紧接着算插入位置时要拿落点去问"那一行是谁"(`dataSource.itemIdentifier(for:)`),
问到的就成了错的行 ——
**算成原位就表现为"拖了没反应",算偏就表现为"落到第三行"**。
两个看起来无关的症状,其实是同一个根因。

**修法**:把那段挪到函数最后(用 `defer`,好覆盖所有 `return` 分支)。

**同一类坑当天又踩了一次**(2026-07-23,`reconfigureItems`):
为修「按编辑后勾选圈时有时无」加了 `collectionView.reconfigureItems(at: 可见的行)`。
它本身是对的,但**退出编辑模式有时是"刚搬完源之后自动触发"的**
(`performMove` 结束会退出编辑模式)—— 那一刻列表和数据源还没对齐,
`indexPathsForVisibleItems` 里混着已经不存在的行 → 撞断言崩溃。
修法两道保险:**延到下一轮 runloop** + **只挑数据源确实认得的行**。

**教训**:
- **拖放、动画、批量更新这类 API 背后都有状态机,"在哪一步调"和"调什么"同等重要。**
  判断标准很实用:**这个调用会不会改动我接下来还要读的东西?** 会,就必须往后放。
  这次要读的是"落点那一行是谁",而 `coordinator.drop` 正好会动它。
- **反过来也要问:我调用它的这一刻,数据稳定吗?**
  `reconfigureItems` / `performBatchUpdates` / `apply` 这些都要求
  "collectionView 看到的世界"和"数据源里的世界"是一致的。
  凡是可能在**数据刚变完**就被调到的路径(尤其是异步搬运的收尾回调里),
  都得延后一轮 + 校验有效性。
- **同一个机制崩到第二次,就别再加保险了 —— 把它拿掉。**
  上面那个 `reconfigureItems` 我先加了"延后一轮",又加了"过滤无效行",
  但它存在的理由其实很单薄:只为了让文件夹的展开三角跟着编辑模式换个样式。
  退一步看才发现**根本不用换** —— 勾选圈和展开三角各管各的点击,天然不冲突
  (点圈=勾选,点整行=展开)。**外观不再依赖编辑状态,那次刷新就整个不需要了。**
  一个需要不断打补丁才能不崩的机制,通常说明它本来就不该存在。

**⚠️ 收到崩溃报告时,先核对二进制指纹再动手改。**
这轮有一份报告其实来自**修复前的版本**(用户在新版装上之前又试了一次)。
崩溃报告里 `usedImages` 有 `NetNewsWire.debug.dylib` 的 UUID,
和当前装的一比就知道是不是同一份:
```bash
A=$(xcrun simctl get_app_container booted com.ranchero.NetNewsWire.iOS-DEBUG app)
dwarfdump --uuid "$A/NetNewsWire.debug.dylib"
```
不核对就照着旧崩溃改,很容易"修一个已经修好的问题",还可能因此引入新问题。
- **一个改动同时引出两个看似无关的症状时,先假设它们同源。**
  这次"排序无效"和"落点偏两行"看着是两个 bug,实际是同一行代码引起的 ——
  分别去修只会越修越乱。
- ⚠️ **这类时序问题离线模拟不出来**(依赖 UIKit 内部状态),
  L63 那套"抽出逻辑跑模拟"在这里帮不上忙 —— 只能靠用户实测。
  所以改动这类 API 时**更要控制单次改动量**:这次是在"拖动功能已经好用"之后
  为了美化动画又动了一次,结果把好用的部分弄坏了。
  **锦上添花的改动,风险不比修 bug 低。**

## 顶栏「渐变透明」问题的现状小结(2026-07-23,未完全解决,交接给后续)

用户想要文章内容页 / 文章列表页的顶栏都做成订阅列表页那种"渐变透明毛玻璃"。折腾多轮未达满意,
**当前落在「不出 bug 但用户觉得不够透」的状态**。已试方案与各自的坑,后续别重复踩:

| 方案 | 结果 |
|---|---|
| 文章内容页 `configureWithDefaultBackground` + trait 监听重建 | 深色发浅 —— registerForTraitChanges 在 UIPageViewController 子树不触发(L59) |
| 文章内容页 全 nil(系统默认毛玻璃) | 深浅自适应、不出 bug,**但偏实,用户嫌不够透**。后查明:**全 nil 永远不会有透明态**(L61) |
| 文章内容页 scrollEdge 完全透明(**当时**) | 露出背后 WKWebView 底,浅色模式顶栏变黑(L60)—— 病根是**网页在自己画底** |
| 文章列表页停靠 scrim: chrome→thin→ultraThin material | 越换越透,但用户仍觉不够;疑似"停靠态顶栏背后是列表顶部空白,毛玻璃无内容可透" |

**2026-07-23 下午:文章内容页已按"纸色归 UIKit + 显式两态"重做,待用户验收。**
上面那张表里的两条死路,其实是**同一个死结的两半**,拆开就都通了:
- 「透明会变黑」的病根**不是透明**,是**网页在自己画底**(深浅色自成一套)。
  → 把纸色底的所有权收归 UIKit(网页 `background: transparent`,
  `WebViewController` 的 view 铺 `AppAppearance.paperBackground` 动态色),透明就安全了。
- 「毛玻璃太实」的病根**不是材质**,是**普通标题栏 scrollEdge=nil 会回落成毛玻璃**(L61),
  根本没有透明那一态可言。→ 显式设透明 scrollEdge + `setContentScrollView` 把线接上。

三件事(纸色归 UIKit / scrollEdge 显式透明 / setContentScrollView)**是一套,缺一不可**:
少第一件 → 复现 L60 的黑顶栏;少后两件之一 → 毛玻璃常驻,和以前一样实。
